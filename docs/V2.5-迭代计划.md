### R1: Cron Error 修复（建议 1-2 天）

**目标**：4 个 Cron 任务全部修复，通过验证测试。

**任务分解**：
1. 定位各 Cron 任务配置文件/代码位置
2. P0-1: 找到 Edit 模式代码，改为 Write overwrite
3. P0-2: 定位 QMD 增量索引同步逻辑，补全 `to` 参数
4. P0-3: 梳理 channel 配置，明确多 channel 消歧规则
5. P0-4: 修正 wecom target 配置
6. 端到端验证：触发各 Cron 任务，确认无报错

**交付**：`git commit` + 验证报告

---

### R2: 性能优化 + 抖音变现报告（建议 3-4 天）

**目标**：核心链路响应时间达标，业务报告恢复自动生成。

**P2-1 SQL Guard 优化**：
- 建立 SQL Guard benchmark 测试
- 定位性能瓶颈（LLM 调用？正则匹配？DB 查询？）
- 优化至 < 50ms

**P2-2 图谱查询优化**：
- 建立图谱查询 benchmark
- 定位瓶颈（Chroma 查询？关系遍历？）
- 优化至 < 500ms

**P1-1 抖音变现报告**：
- 定位报告生成逻辑断点
- 修复数据获取/模板渲染问题
- 恢复每日/每周自动报告生成

**交付**：性能基线报告 + 报告生成验证

---

## 四、风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Cron 错误根因复杂 | 可能超出预估时间 | R1 预留缓冲，拆分细粒度验证 |
| 抖音报告数据源中断 | 报告无法生成 | 先确认数据源可用性再开始 |
| 性能优化触及底层架构 | 改动风险高 | 先做 benchmark，定位瓶颈再动手 |
| 竞争分析需要主观判断 | 结论争议性 | 基于公开数据，注明假设前提 |
