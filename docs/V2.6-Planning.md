# Javis-DB-Agent V2.6 规划方案

> 版本: 2.6.0-draft | 日期: 2026-04-01 | 架构师: 道衍

---

## 一、背景与目标

### 1.1 版本背景

当前版本 V2.5 已实现**本地数据库自动发现与纳管**能力，核心架构采用 **Orchestrator 单一编排器 + 多个专业 Agent** 的单层串行架构。

随着功能增长，现有架构面临以下挑战：

| 问题 | 现状 | 影响 |
|------|------|------|
| Agent 执行串行化 | `_execute_plan` 顺序调用各 Agent | 诊断/分析类任务耗时长 |
| 无生命周期钩子 | 无插件化扩展机制 | 难以接入监控/审计/第三方系统 |
| 安全护栏可绕过 | 审批流程依赖调用方主动检查 | 高风险操作存在绕过风险 |
| 无跨会话记忆 | 每次会话从头开始 | 无法学习用户偏好和 DBA 习惯 |
| 无可观测性 | 无结构化埋点和指标 | 线上问题排查困难 |

### 1.2 借鉴来源

本规划借鉴 **Claude Code** 的多智能体能力，但**不直接使用其代码**，而是学习其设计思路并适配到 Javis DB Agent 的数据库运维场景。

| Claude Code 能力 | 核心设计 | Javis-DB-Agent 适配方向 |
|------------------|----------|------------------------|
| Coordinator 编排 | 动态任务分解 + 智能调度 | 并行 Agent 执行引擎 |
| autoDream 记忆 | fork 后台进程 + 持续学习 | 跨会话记忆系统 |
| SafetyCheck | Step 7 不可绕过安全检查 | 强制审批门卫 + PreToolUse Hook |
| Hook 扩展体系 | 生命周期钩子 + 插件化 | 事件驱动 Hook 系统 |
| code-review 多Agent | 4 Agent 并行 + 置信度评分 | 并行诊断 + 置信度结果聚合 |

### 1.3 V2.6 目标

**V2.6 核心目标**：在 V2.5 基础上，引入**事件驱动 Hook 系统**、**并行 Agent 执行引擎**、**不可绕过安全护栏**、**可观测性框架**，为未来 V3.0 的多Agent协作（Coordinator）奠定架构基础。

## 四、详细设计

### 4.1 F1: 事件驱动 Hook 系统 [P0]

#### 4.1.1 设计思路

借鉴 Claude Code hookify 插件的**声明式规则 + 事件驱动**机制，为 Javis-DB-Agent 建立生命周期 Hook 系统。

#### 4.1.2 核心概念

```python
# Hook 事件类型
class HookEvent(Enum):
    # Agent 生命周期
    AGENT_BEFORE_INVOKE = "agent:before_invoke"      # Agent 调用前
    AGENT_AFTER_INVOKE = "agent:after_invoke"         # Agent 调用后
    AGENT_ERROR = "agent:error"                        # Agent 执行错误

    # Tool 生命周期（类比 Claude Code PreToolUse/PostToolUse）
    TOOL_BEFORE_EXECUTE = "tool:before_execute"       # Tool 执行前
    TOOL_AFTER_EXECUTE = "tool:after_execute"         # Tool 执行后
    TOOL_ERROR = "tool:error"                         # Tool 执行错误

    # SQL 专用（增强现有 sql_guard）
    SQL_BEFORE_GUARD = "sql:before_guard"              # SQL 护栏检查前
    SQL_AFTER_GUARD = "sql:after_guard"               # SQL 护栏检查后
    SQL_DDL_DETECTED = "sql:ddl_detected"              # DDL 语句检测

    # 审批生命周期
    APPROVAL_REQUESTED = "approval:requested"         # 发起审批
    APPROVAL_APPROVED = "approval:approved"            # 审批通过
    APPROVAL_REJECTED = "approval:rejected"            # 审批拒绝

    # 会话生命周期
    SESSION_START = "session:start"                   # 会话开始
    SESSION_END = "session:end"                       # 会话结束

    # 自定义业务事件
    ALERT_TRIGGERED = "alert:triggered"                # 告警触发
    INSPECT_REPORT_GENERATED = "inspect:report_generated" # 巡检报告生成
```

#### 4.1.3 Hook 规则配置格式

参考 Claude Code hookify 的 Markdown 配置格式，设计 Javis-DB-Agent 的 Hook 规则：

```yaml
# .javis/hooks.d/ddl-confirm-hook.yaml
---
name: ddl-confirm-hook
enabled: true
event: sql:ddl_detected
action: block  # block | warn | log
conditions:
  - field: sql_statement
    operator: regex_match
    pattern: (DROP|TRUNCATE|ALTER\s+TABLE)\s+
  - field: risk_level
    operator: gte
    value: L3
---

⚠️ **DDL 操作确认**

检测到高风险 DDL 操作：
{{ sql_statement }}

请确认是否继续执行。
```

#### 4.1.4 HookEngine 架构

```
HookEngine
├── HookRegistry          # 规则注册表
│   ├── _hooks: dict[HookEvent, list[HookRule]]
│   ├── register(rule, handler)
│   ├── unregister(name)
│   └── list_hooks(event)
│
├── RuleEngine            # 规则评估引擎（参考 hookify）
│   ├── evaluate_rules(rules, context) → HookResult
│   ├── _match_conditions(rule, context) → bool
│   └── _compute_action(rules) → BlockingRule | WarningRule
│
├── HookContext           # Hook 执行上下文
│   ├── event: HookEvent
│   ├── payload: dict     # 事件数据（SQL 语句、Agent 名称等）
│   ├── session_id: str
│   └── user_id: str
│
└── emit(event, payload)  # 触发 Hook
    ├── load matching rules
    ├── evaluate with RuleEngine
    └── execute actions (block/warn/log)
```

#### 4.1.5 与现有模块集成

```python
# 在 tool_registry.py 的 execute 方法中集成
class ToolRegistry:
    async def execute(self, tool_name: str, params: dict, context: dict):
        # 1. TOOL_BEFORE_EXECUTE Hook
        hook_result = await HookEngine.emit(
            HookEvent.TOOL_BEFORE_EXECUTE,
            {"tool_name": tool_name, "params": params, "context": context}
        )
        if hook_result.blocked:
            return ToolResult(success=False, error="Hook blocked: " + hook_result.message)

        # 2. 实际执行
        result = await self._execute_impl(tool_name, params)

        # 3. TOOL_AFTER_EXECUTE Hook
        await HookEngine.emit(
            HookEvent.TOOL_AFTER_EXECUTE,
            {"tool_name": tool_name, "result": result, "context": context}
        )
        return result
```

#### 4.1.6 价值与意义

| 维度 | 说明 |
|------|------|
| **插件化** | 第三方可通过 Hook 接入监控/告警/审计，无需修改核心代码 |
| **可组合** | 多个 Hook 可监听同一事件，支持链式处理 |
| **声明式** | YAML 配置即可添加新 Hook，降低使用门槛 |
| **基础设施** | F3（安全护栏）和 F4（可观测性）都依赖此系统 |

### 4.2 F2: 并行 Agent 执行引擎 [P0]

#### 4.2.1 设计思路

将当前 `_execute_plan` 的**串行执行**改造为**并行执行 + 置信度加权聚合**，借鉴 Claude Code code-review 插件的 4 Agent 并行 + 置信度评分模式。

#### 4.2.2 场景分类

| 场景 | 执行模式 | 说明 |
|------|----------|------|
| 告警诊断 (DIAGNOSE) | 并行: DiagnosticAgent + RiskAgent | 多角度并行诊断 |
| SQL 分析 (SQL_ANALYZE) | 并行: SQLAnalyzerAgent + RiskAgent | 性能+风险并行 |
| 会话分析 (ANALYZE_SESSION) | 并行: SessionAnalyzerAgent + PerformanceAgent | 状态+性能并行 |
| 根因分析 (ROOT_CAUSE) | 并行: AlertAgent + DiagnosticAgent + RiskAgent | 三维度并行 |
| 健康巡检 (INSPECT) | 并行: InspectorAgent + CapacityAgent | 状态+容量并行 |
| 风险评估 (RISK_ASSESS) | 并行: RiskAgent + SQLAnalyzerAgent + SessionAnalyzerAgent | 多Agent交叉验证 |
| 其他 | 串行（原逻辑） | 无需并行的场景保持兼容 |

#### 4.2.3 ParallelAgentExecutor 设计

```python
class ParallelAgentExecutor:
    """
    并行 Agent 执行器

    核心能力：
    1. 根据 Intent 类型决定并行/串行策略
    2. asyncio.gather 并发执行多个 Agent
    3. 置信度评分 + 加权聚合结果
    4. 超时控制 + 部分失败处理
    """

    # 并行策略配置
    PARALLEL_INTENTS = {
        Intent.DIAGNOSE: ["diagnostic", "risk"],
        Intent.SQL_ANALYZE: ["sql_analyzer", "risk"],
        Intent.ANALYZE_SESSION: ["session_analyzer", "performance"],
        Intent.ROOT_CAUSE: ["alert", "diagnostic", "risk"],
        Intent.INSPECT: ["inspector", "capacity"],
        Intent.RISK_ASSESS: ["risk", "sql_analyzer", "session_analyzer"],
    }

    # Agent 置信度基线（可配置）
    AGENT_CONFIDENCE = {
        "diagnostic": 0.9,
        "risk": 0.85,
        "sql_analyzer": 0.9,
        "session_analyzer": 0.85,
        "performance": 0.8,
        "alert": 0.85,
        "capacity": 0.8,
        "inspector": 0.9,
        "reporter": 0.95,
    }

    async def execute(
        self,
        intent: Intent,
        goal: str,
        context: dict,
        agents: list[BaseAgent]
    ) -> AgentResponse:
        """
        执行 Agent

        Args:
            intent: 用户意图
            goal: 任务目标
            context: 执行上下文
            agents: 要执行的 Agent 列表

        Returns:
            聚合后的 AgentResponse（含置信度）
        """
        # 1. 判断是否并行
        parallel_agents = self.PARALLEL_INTENTS.get(intent, [])
        should_parallel = any(a.name in parallel_agents for a in agents) and len(agents) > 1

        if not should_parallel:
            # 串行执行（兼容非并行场景）
            return await self._execute_sequential(agents, goal, context)

        # 2. 并行执行
        tasks = [
            agent.process(goal, context)
            for agent in agents
        ]
        results: list[AgentResponse] = await asyncio.gather(
            *tasks,
            return_exceptions=True
        )

        # 3. 过滤异常
        valid_results = [
            r for r in results
            if isinstance(r, AgentResponse) and r.success
        ]

        if not valid_results:
            return self._build_error_response(results)

        # 4. 置信度加权聚合
        return self._aggregate_with_confidence(valid_results, intent)

    def _aggregate_with_confidence(
        self,
        results: list[AgentResponse],
        intent: Intent
    ) -> AgentResponse:
        """
        置信度加权聚合

        参考 Claude Code code-review 的置信度评分：
        - 每条结果有置信度分 (0-1)
        - 低于阈值的结论被过滤
        - 高置信度结论加权合并
        """
        # 计算每条结果的综合分数
        scored_results = []
        for r in results:
            agent_name = r.metadata.get("agent", "unknown")
            base_confidence = self.AGENT_CONFIDENCE.get(agent_name, 0.7)
            # 根据执行时间调整（过快可能是粗略判断，过慢可能是复杂问题）
            time_factor = min(r.execution_time_ms / 5000, 1.0)  # 5s 封顶
            confidence = base_confidence * (0.8 + 0.2 * time_factor)
            scored_results.append((r, confidence))

        # 过滤低置信度（< 0.6）
        high_confidence = [(r, c) for r, c in scored_results if c >= 0.6]

        if not high_confidence:
            # 无高置信度结果，返回最高分
            best = max(scored_results, key=lambda x: x[1])
            return best[0]

        # 加权合并内容
        aggregated_content = self._merge_content(high_confidence)

        return AgentResponse(
            success=True,
            content=aggregated_content,
            metadata={
                "aggregation": "confidence_weighted",
                "scores": {r.metadata.get("agent"): c for r, c in scored_results},
                "used_count": len(high_confidence),
                "total_count": len(results),
            }
        )
```

#### 4.2.4 与 Orchestrator 集成

```python
# 在 orchestrator.py 中替换 _execute_plan
class OrchestratorAgent(BaseAgent):
    def __init__(self):
        super().__init__()
        self._parallel_executor = ParallelAgentExecutor()  # 新增

    async def _execute_plan(self, plan: list[dict], context: dict) -> list[AgentResponse]:
        # 意图从 context 中获取（由 _recognize_intent 填充）
        intent = context.get("recognized_intent", Intent.GENERAL)

        # 使用并行执行器
        agents = [self.get_agent(step["agent"]) for step in plan]

        return [result]  # 保持返回 list[AgentResponse] 兼容
```

### 4.3 F3: 不可绕过安全护栏 [P0]

#### 4.3.1 设计思路

将现有的**审批流程**改造为**不可绕过 Gate**，并在 Tool 执行前增加 **PreToolUse Hook** 检查，形成双重保障。

借鉴 Claude Code 的 SafetyCheck 机制：
- Claude Code: Step 7 安全检查不可绕过，强制人工确认
- Javis-DB-Agent: 所有 L4/L5 操作必须经过审批 Gate，无法跳过

#### 4.3.2 当前审批流程的问题

```
当前流程（可绕过）：
用户请求 → PolicyEngine.check() → 需要审批 → [如果调用方不检查 ApprovalStatus，直接执行 Tool] → 风险！

问题：PolicyEngine 返回 approval_required=True，但实际执行权在 Tool 层
如果 Tool 实现者忘记检查 approval_required，审批就被绕过了
```

#### 4.3.3 强制审批 Gate 架构

```
┌─────────────────────────────────────────────────────────────┐
│                 SafetyGuardRail (不可绕过)                   │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ToolRegistry.execute()                                      │
│       │                                                      │
│       ├──▶ HookEngine.emit(TOOL_BEFORE_EXECUTE)            │
│       │         │                                            │
│       │         └──▶ DDL Hook: 检查 DDL 语句（block/warn）   │
│       │         │                                            │
│       │         └──▶ Risk Hook: 检查风险级别                  │
│       │         │                                            │
│       ├──▶ SafetyGuardRail.enforce(tool, risk_level)       │
│       │         │                                            │
│       │         ├──▶ L1-L3: 直接放行                          │
│       │         ├──▶ L4: 强制单签审批（不可绕过）              │
│       │         │         └──▶ 审批通过令牌 (token)           │
│       │         │              └──▶ 写入执行上下文            │
│       │         │                                            │
│       │         └──▶ L5: 强制双人审批（不可绕过）             │
│       │                   └──▶ 双令牌验证                    │
│       │                        └──▶ 写入执行上下文            │
│       │                                                      │
│       ├──▶ 执行 Tool（只有持有有效审批令牌的才能执行）        │
│       │                                                      │
│       └──▶ HookEngine.emit(TOOL_AFTER_EXECUTE)              │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

#### 4.3.4 SafetyGuardRail 实现

```python
class SafetyGuardRail:
    """
    不可绕过安全护栏

    核心原则：
    1. ToolRegistry.execute() 必须经过此 Gate
    2. L4/L5 操作无有效审批令牌则抛出异常（不是返回错误，是异常）
    3. 审批令牌存储在执行上下文 context["approval_tokens"] 中
    4. Tool 执行层不持有审批判断权，只持有令牌验证权
    """

    def __init__(self, approval_gate: ApprovalGate):
        self._approval_gate = approval_gate

    async def enforce(
        self,
        tool_name: str,
        risk_level: RiskLevel,
        context: dict
    ) -> None:
        """
        强制执行安全检查

        Args:
            tool_name: 工具名称
            risk_level: 风险级别
            context: 执行上下文（含 approval_tokens）

        Raises:
            ApprovalRequiredError: L4/L5 操作无有效令牌
        """
        # L1-L3 直接放行
        if risk_level in (RiskLevel.L1_READ, RiskLevel.L2_DIAGNOSE, RiskLevel.L3_LOW_RISK):
            return

        # L4: 单签审批
        if risk_level == RiskLevel.L4_MEDIUM:
            token = context.get("approval_tokens", {}).get(f"{tool_name}:L4")
            if not token:
                # 发起审批请求（同步等待）
                approval_result = await self._approval_gate.request_approval(
                    tool_name=tool_name,
                    risk_level="L4",
                    requester=context.get("user_id", "unknown"),
                    params=context.get("params", {}),
                    timeout=300  # 5分钟超时
                )
                if not approval_result.success:
                    raise ApprovalRequiredError(
                        f"L4 操作 {tool_name} 需要审批，当前未通过"
                    )
                # 写入令牌
                context.setdefault("approval_tokens", {})[f"{tool_name}:L4"] = approval_result.token

            return

        # L5: 双人审批
        if risk_level == RiskLevel.L5_HIGH:
            token_l5 = context.get("approval_tokens", {}).get(f"{tool_name}:L5")
            if not token_l5:
                approval_result = await self._approval_gate.request_dual_approval(
                    tool_name=tool_name,
                    risk_level="L5",
                    requester=context.get("user_id", "unknown"),
                    params=context.get("params", {}),
                    timeout=600
                )
                if not approval_result.success:
                    raise ApprovalRequiredError(
                        f"L5 操作 {tool_name} 需要双人审批，当前未通过"
                    )
                context.setdefault("approval_tokens", {})[f"{tool_name}:L5"] = approval_result.token

            return

    def verify_token(self, tool_name: str, risk_level: RiskLevel, context: dict) -> bool:
        """验证审批令牌（Tool 执行层调用）"""
        token_key = f"{tool_name}:{risk_level.value}"
        return context.get("approval_tokens", {}).get(token_key) is not None
```

## 五、迭代分解

### 5.1 迭代总览

```
V2.6
├── R1: Hook 基础框架 + 安全护栏 (2-3天)
│       ↓
├── R2: 并行 Agent 执行引擎 (2天)
│       ↓
├── R3: 可观测性框架 (2天)
│       ↓
└── V2.6 稳定版发布
```

### 5.2 R1: Hook 基础框架 + 安全护栏

**目标**：建立事件驱动 Hook 基础设施，实现不可绕过安全护栏

**交付物**：
1. `src/gateway/hooks/` 目录及核心模块
2. `HookEngine` + `HookRegistry` + `RuleEngine`
3. 5 个核心 Hook 事件实现
4. `SafetyGuardRail` 强制审批 Gate
5. DDL 检测 Hook 规则
6. Hook 配置格式文档

**依赖关系**：无（纯新增）

**验收标准**：
- [ ] `HookEngine.emit()` 可正常触发事件
- [ ] DDL 语句（DROP/TRUNCATE）被 Hook 拦截并要求审批
- [ ] L4/L5 操作无法绕过审批 Gate
- [ ] Hook 规则可通过 YAML 配置启用/禁用

---

### 5.3 R2: 并行 Agent 执行引擎

**目标**：将 Agent 执行从串行改为并行，引入置信度加权聚合

**交付物**：
1. `src/agents/executor.py` - `ParallelAgentExecutor`
2. 意图-并行策略配置
3. 置信度评分 + 加权聚合算法
4. 与 Orchestrator 的集成代码
5. 向后兼容处理（非并行意图保持串行）

**依赖关系**：
- 依赖 R1 的 Hook 基础设施（用于并行执行的埋点）
- 可独立于 R1 开发，但需 R1 Hook 系统作为基础设施

**验收标准**：
- [ ] DIAGNOSE 场景下 DiagnosticAgent + RiskAgent 并行执行
- [ ] 并行执行耗时 < 串行执行耗时（相同硬件）
- [ ] 置信度 < 0.6 的结果被过滤
- [ ] 单 Agent 场景无性能退化

---

### 5.4 R3: 可观测性框架

**目标**：基于 Hook 系统实现零侵入埋点，提供 Traces/Metrics/Logs

**交付物**：
1. `src/observability/` 目录及核心模块
2. `TraceManager` - 链路追踪
3. `MetricsCollector` - 指标收集
4. `LogEmitter` - 结构化日志
5. `TelemetryExporter` - 导出器（Console + File + Prometheus）
6. Dashboard 监控页面（可选）

**依赖关系**：
- 强依赖 R1 的 Hook 系统（埋点事件源）

**验收标准**：
- [ ] Agent/Tool 执行自动生成 Trace
- [ ] 关键指标（调用次数、耗时、错误率）可查询
- [ ] 日志包含 Trace ID，可关联请求
- [ ] 导出到 Prometheus 格式可正常采集

| 并行执行结果不确定 | 低 | 中 | 置信度评分 + 串行 fallback |
| 审批 Gate 破坏现有流程 | 中 | 高 | 向后兼容 + 配置开关 |
