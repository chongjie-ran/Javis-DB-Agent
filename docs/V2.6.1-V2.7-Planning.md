## 二、V2.6.1 规划

### 2.1 Token TTL 机制

#### 2.1.1 问题分析

**现状**：
```python
# guard_rail.py - _enforce_l4/l5
tokens[token_key] = result.request_id  # 只存 token，无过期时间
```

`approval_tokens` 写入 context 后**永不过期**。意味着：
- 用户审批通过后，同一 tool_name + risk_level 在会话内永久有效
- 审批通过后如果 SQL/参数变了，实际执行的内容可能与审批内容不同
- 高风险操作的审批令牌无时效限制，存在安全隐患

**场景示例**：
```
用户审批 L4 操作 "DROP TABLE orders"
→ 审批通过，令牌写入 approval_tokens["execute_sql:L4"]
→ 用户修改参数，改成 "DROP TABLE users"（更危险）
→ 直接执行，因为令牌存在
```

**实现价值**：⭐⭐⭐⭐ 中高
- 防止审批内容与执行内容不一致（参数漂移）
- 符合安全最小权限原则
- 防止令牌被盗用后的无限期使用

**实现难度**：⭐⭐ 简单
- 只需在 token 存储时附加 timestamp
- 验证时检查是否在 TTL 窗口内
- 改动点集中：`SafetyGuardRail` 和 `ApprovalGate`

#### 2.1.2 技术方案

**方案设计**：采用 TTL + 内容哈希双重校验

```python
# 方案：Token 结构升级
@dataclass
class ApprovalToken:
    request_id: str
    tool_name: str
    risk_level: str
    params_hash: str       # params 的 SHA256，检测参数漂移
    created_at: float     # 创建时间戳
    expires_at: float     # 过期时间戳
    approver: str         # 审批人（用于审计）
```

**TTL 配置**：
```yaml
# config.yaml
approval:
  token_ttl_seconds: 600  # 默认10分钟
  # L4: 10分钟，L5: 5分钟（L5更危险，TTL更短）
  l4_ttl_seconds: 600
  l5_ttl_seconds: 300
```

**验证逻辑**：
```python
def _validate_token(self, token_key: str, params: dict, tokens: dict) -> bool:
    token_data = tokens.get(token_key)
    if not token_data:
        return False

    # 1. 检查 TTL
    if time.time() > token_data.expires_at:
        del tokens[token_key]  # 过期删除
        return False

    # 2. 检查参数漂移（防止审批内容与执行内容不一致）
    current_hash = self._hash_params(params)
    if current_hash != token_data.params_hash:
        logger.warning(f"Params drift detected for {token_key}")
        return False

    return True
```

**改动范围**：
| 文件 | 改动 |
|------|------|
| `src/gateway/approval.py` | `ApprovalRequest` 增加 `params_hash` 字段；`get_status` 返回时携带 |
| `src/security/guard_rail.py` | `ApprovalToken` dataclass；`_validate_token` 方法；`_enforce_l4/l5` 改用新结构 |

#### 2.1.3 任务分解

- [ ] **T1.1**: 在 `approval.py` 的 `ApprovalRequest` 增加 `params_hash: str` 字段
- [ ] **T1.2**: 在 `ApprovalGate.request_approval()` 中计算 params_hash 并存储
- [ ] **T1.3**: 在 `SafetyGuardRail` 中定义 `ApprovalToken` dataclass（request_id, tool_name, risk_level, params_hash, created_at, expires_at, approver）
- [ ] **T1.4**: 实现 `_validate_token()` 方法（TTL + params_hash 校验）
- [ ] **T1.5**: 重构 `_enforce_l4` 和 `_enforce_l5` 使用新 token 结构和验证逻辑
- [ ] **T1.6**: 新增配置项 `approval.l4_ttl_seconds` / `approval.l5_ttl_seconds`
- [ ] **T1.7**: 单元测试：token 过期、参数漂移检测

### 2.2 Hook MODIFY 动作实现

#### 2.2.1 问题分析

**现状**：
```python
# rule_engine.py - _execute_action
elif action == HookAction.MODIFY:
    # MODIFY 动作由 handler 处理，这里仅记录
    logger.debug(f"Hook modify: {rule.name}")
```

`HookAction.MODIFY` 已在 `hook_rule.py` 定义，但 `RuleEngine._execute_action` **只打日志，不执行任何 payload 修改**。

MODIFY 动作的价值在于：可以在 Tool 执行前**动态修改参数**，例如：
- 敏感字段脱敏（将 `password` 字段替换为 `****`）
- SQL 语句重写（自动加 LIMIT 防止全表扫描）
- 参数边界修正（将 `LIMIT 100000` 改为 `LIMIT 1000`）

**实现价值**：⭐⭐⭐ 中
- 完善 Hook 动作体系（目前 BLOCK/WARN/LOG 都已实现，唯独 MODIFY 空转）
- 为 SQL 防护提供更强的手段（自动重写危险 SQL）
- 实现后可替代部分硬编码的安全检查逻辑

**实现难度**：⭐⭐⭐ 中
- MODIFY 需要修改 `context.payload`，需要明确修改语义（替换字段？追加字段？删除字段？）
- 规则配置格式需要扩展（当前 YAML 规则只有 conditions + action + message）
- 需要定义清晰的修改语义，防止规则之间冲突

#### 2.2.2 技术方案

**修改语义定义**：
```python
class ModifyOperation(str, Enum):
    REPLACE = "replace"    # 替换字段值
    REDACT = "redact"      # 脱敏（替换为占位符）
    ADD = "add"           # 新增字段
    REMOVE = "remove"     # 删除字段
    CLAMP = "clamp"       # 限制数值范围（如 LIMIT 上限）
```

**扩展 HookRule 配置格式**：
```yaml
# .javis/hooks.d/sql-safe-modify.yaml
---
name: sql-limit-modify
enabled: true
event: tool:before_execute
action: modify
conditions:
  - field: tool_name
    operator: eq
    value: execute_sql
  - field: params.statement
    operator: regex_match
    pattern: SELECT\s+.+\s+FROM\s+\w+
modify:
  - operation: clamp
    field: params.limit
    min: 1
    max: 1000
    default: 1000
---
```

**RuleEngine 改造**：
```python
# rule_engine.py - _execute_action 新增 MODIFY 处理
elif action == HookAction.MODIFY:
    if rule.modify_ops:
        for op in rule.modify_ops:
            context = self._apply_modify(context, op)
    logger.debug(f"Hook modify applied: {rule.name}")

def _apply_modify(self, context: HookContext, op: ModifyOperation) -> HookContext:
    """应用修改操作到 payload"""
    field = op.field  # 如 "params.limit" 或 "payload.sql_statement"
    value = op.value

    if op.operation == ModifyOperation.REPLACE:
        context.set(field, value)
    elif op.operation == ModifyOperation.REDACT:
        context.set(field, op.redact_with or "****")
    elif op.operation == ModifyOperation.CLAMP:
        current = context.get(field, op.default)
        clamped = max(op.min, min(current, op.max))
        context.set(field, clamped)
    elif op.operation == ModifyOperation.REMOVE:
        parts = field.split(".")
        payload = context.payload
        # 递归找到父级 dict 并删除
        ...
    return context
```

**HookRule 扩展字段**：
```python
@dataclass
class ModifyOperation:
    operation: ModifyOperation
    field: str           # 字段路径
    value: Any = None   # REPLACE/ADD 的值
    redact_with: str = "****"  # REDACT 替换值
    min: int = None     # CLAMP 下限
    max: int = None     # CLAMP 上限
    default: Any = None # CLAMP 默认值

@dataclass
class HookRule:
    # ... 现有字段 ...
    modify_ops: list[ModifyOperation] = field(default_factory=list)  # 新增
```

## 三、V2.7 规划

### 3.1 Webhook/Callback 替代审批轮询

#### 3.1.1 问题分析

**现状**：
```python
# guard_rail.py - _request_approval
while True:
    status_resp = await gate.get_status(request_id)
    if status_resp.status != ApprovalStatus.PENDING:
        ...
    await asyncio.sleep(min(poll_interval, timeout - elapsed))  # 2秒轮询
```

问题：
- **资源浪费**：每个待审批请求占用一个 asyncio 任务 + 2秒间隔的 sleep
- **响应延迟**：最长2秒延迟才能感知审批结果
- **扩展性差**：并发审批请求多时，轮询任务堆积

**当前架构**：
```
SafetyGuardRail              ApprovalGate
    │                              │
    ├─ request_approval() ───────▶ 写入 _requests[request_id]
    │                              │
    ├─ get_status() ──────────────▶ 读取 _requests[request_id]   ← 2秒轮询
    │                              │
    ◀── asyncio.sleep(2) ──────────┘
```

**目标架构**（Webhook 模式）：
```
SafetyGuardRail              ApprovalGate              外部审批系统
    │                              │                        │
    ├─ request_approval() ───────▶│ 写入 _requests          │
    │                              │                         │
    │   [注册 callback URL]        │                         │
    │                              ├─ 转发到外部审批系统       │
    │                              │                         │
    │   [注册 asyncio.Event]       │◀── approval_webhook()───┤ 审批完成
    │                              │                         │
    ◀── Event.set() (立即唤醒)      │                         │
```

**实现价值**：⭐⭐⭐⭐ 高
- 消除2秒轮询延迟（审批结果几乎是即时的）
- 减少 asyncio 任务堆积（特别是高并发场景）
- 为接入飞书/企微审批系统奠定基础（ webhook 是标准集成方式）
- 符合事件驱动架构方向（V2.6 已引入 Hook 系统）

**实现难度**：⭐⭐⭐⭐ 较高
- 需要改造 `ApprovalGate` 的状态通知机制
- 需要处理 webhook 注册、去重、超时
- 需要保持向后兼容（轮询模式作为 fallback）
- 外部审批系统 webhook 的接入规范需定义

#### 3.1.2 技术方案

**核心设计**：

```python
class ApprovalGate:
    def __init__(self, ...):
        # 现有字段
        self._requests: dict[str, ApprovalRequest] = {}
        self._events: dict[str, asyncio.Event] = {}
        # 新增字段
        self._webhook_callbacks: dict[str, Callable] = {}  # request_id -> callback
        self._use_webhook: bool = False  # 配置开关

    async def request_approval(self, ..., webhook_callback: Callable = None):
        # 现有逻辑...
        request = ApprovalRequest(...)
        self._requests[request_id] = request
        self._events[request_id] = asyncio.Event()

        # 新增：注册 webhook callback
        if webhook_callback:
            self._webhook_callbacks[request_id] = webhook_callback

        return ApprovalRequestResult(success=True, request_id=request_id)
```

**Webhook 路由**（新增 API 路由）：
```python
# api/approval_webhook.py
@router.post("/approval/webhook")
async def approval_webhook(request: WebhookPayload):
    """
    外部审批系统回调接口

    外部审批系统（如飞书审批）在审批完成后调用此接口。
    """
    request_id = request.request_id
    action = request.action  # "approve" | "reject"

    gate = get_approval_gate()

    if action == "approve":
        approver = request.approver or "external"
        await gate.approve(request_id, approver, request.comment or "")
    elif action == "reject":
        await gate.reject(request_id, request.approver or "external", request.reason or "")

    # 清理 webhook 回调
    gate._webhook_callbacks.pop(request_id, None)

    return {"status": "ok"}
```

**SafetyGuardRail 改造**：
```python
async def _request_approval(self, ...):
    import asyncio

    # 方案A：WebSocket 实时通知（推荐）
    # 使用 asyncio.Event 作为本地 callback，不再轮询

    approval_event = asyncio.Event()

    async def on_approval_result(result: ApprovalRequestResult):
        # 外部 webhook 触发此回调
        approval_event.set()

    gate = self.approval_gate

    # 注册 callback
    await gate.register_callback(request_id, on_approval_result)

    # 等待 callback 触发或超时
    try:
        await asyncio.wait_for(
            approval_event.wait(),
            timeout=timeout
        )
    except asyncio.TimeoutError:
        return ApprovalRequestResult(success=False, request_id=request_id, error="审批超时")

    # 读取最终状态
    status_resp = await gate.get_status(request_id)
    ...
```

**配置扩展**：
```yaml
# config.yaml
approval:
  # V2.7 新增
  webhook_enabled: true
  webhook_endpoint: "http://localhost:8000/approval/webhook"
  # 轮询 fallback（webhook 不可用时启用）
  poll_fallback_enabled: true
  poll_interval_seconds: 2
```

**向后兼容**：
- 默认 `webhook_enabled: false`，保持 V2.6.1 的轮询行为
- 启用 webhook 后，轮询作为 fallback（webhook 注册失败/超时降级）
- 通过配置开关平滑升级

## 五、风险与依赖

### 5.1 V2.6.1 风险

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| Token TTL 改动破坏现有审批流程 | 低 | 高 | 单元测试覆盖 + 配置默认保持向后兼容 |

|------|------|------|------|
