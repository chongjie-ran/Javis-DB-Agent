### 2.2 发现的Bug

#### Bug 1: 阻断级 - UnboundLocalError（严重度：高）

**位置**：`src/gateway/approval.py` 第 236 行

**问题描述**：
```python
request = ApprovalRequest(
    request_id=request_id,
    ...
    params_hash=request.compute_params_hash(),  # BUG: request 未创建！
)
```

`request.compute_params_hash()` 在 `request` 对象创建之前被调用，导致 `UnboundLocalError`。

**影响**：所有调用 `request_approval()` 的操作都会崩溃，Token TTL 机制完全不可用。

**修复方案**：
```python
# 计算 params_hash 在创建 request 之前
_params_hash = hashlib.sha256(str(sorted(params.items())).encode()).hexdigest()

request = ApprovalRequest(
    request_id=request_id,
    ...
    params_hash=_params_hash,  # 使用预计算的值
)
```

#### Issue 2: 字段路径语义混淆（严重度：低）

**位置**：`src/gateway/hooks/rule_engine.py` `_apply_modify()` 和 `_set_nested()`

**问题描述**：
当 YAML 规则指定 `field: payload.sql` 时，代码会创建嵌套结构 `{'payload': {'sql': ...}}`，而不是修改顶层的 `sql` 键。

这与 HookContext 的设计不一致 - `context.payload` 就是 payload 数据，不是 payload 内部还有一个 payload。

**示例**：
```python
# 错误用法（但用户可能这样写）
rule = HookRule(
    modify_ops=[ModifyOperation(field='payload.sql', value='SELECT 1')]
)
ctx = HookContext(payload={'sql': 'SELECT * FROM users'}, ...)
# 结果：ctx.payload = {'sql': 'SELECT * FROM users', 'payload': {'sql': 'SELECT 1'}}

# 正确用法
rule = HookRule(
    modify_ops=[ModifyOperation(field='sql', value='SELECT 1')]
)
# 结果：ctx.payload = {'sql': 'SELECT 1'}
```

**建议**：
1. 在文档中明确说明 field 路径的语义
2. 或修改代码支持 `payload.` 前缀的特殊处理（跳过 payload 直接访问其内部键）
