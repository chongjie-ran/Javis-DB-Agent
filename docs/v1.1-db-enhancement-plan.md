## 一、现有架构分析

### 1.1 当前架构回顾

```
用户层
  ↓
Agent Gateway（会话管理、工具注册、策略引擎、审计）
  ↓
智能决策层
  ├── OrchestratorAgent（编排）
  ├── DiagnosticAgent（诊断）
  ├── RiskAgent（风险）
  ├── SQLAnalyzerAgent（SQL分析）
  ├── InspectorAgent（巡检）
  └── ReporterAgent（报告）
  ↓
工具执行层（查询工具 + 分析工具 + 行动工具）
  ↓
API层（Mock Javis API / Real Javis API）
  ↓
LLM层（Ollama 本地运行）
```

**当前限制**：
- 工具层使用模拟数据，未真正对接多引擎
- Real API Client 仅针对 MySQL 风格 API
- 缺少 PostgreSQL 的专属工具和路由
- 会话分析、死锁检测能力薄弱
- 容量管理仅有基础表空间查询
- 无智能告警、预测性告警能力

### 1.2 扩展现有架构支持 MySQL/PG

**核心策略：分层解耦**

```
DB Adapter Layer（数据库适配层）
  ├── MySQLAdapter（MySQL 专有实现）
  └── PostgreSQLAdapter（PostgreSQL 专有实现）
  ↓
Unified DB Interface（统一数据库接口）
  ↓
工具层（双引擎兼容）
```

**适配策略**：
- **连接层**：统一连接抽象，引擎特有参数（PG: `port=5432`, MySQL: `port=3306`）
- **元数据查询**：统一 Schema 获取接口，引擎内部翻译
- **指标采集**：统一性能指标模型，各自适配采集方式
- **告警规则**：统一告警类型，各自映射到引擎指标

```
你是 MonitorAgent，负责数据库的实时性能监控和会话分析。

输出格式：
```

### 2.3 AlertAgent 设计

**文件**：`src/agents/alert.py`

**System Prompt**：
```
你是 AlertAgent，负责智能告警分析和根因定位。

角色定义：
- 接收原始告警，进行智能分类和去重
- 基于告警关联规则进行根因分析
- 识别告警模式，预测潜在故障

1. 告警压缩：同一根因的多条告警压缩为一条
```

### Round 12（R1）- 基础设施：双引擎适配层

**目标**：建立 MySQL/PG 统一适配层，支撑后续所有工具扩展

#### 3.1.1 新增数据库适配层

**文件**：`src/db/connector.py`（新增目录）
```
src/db/
  ├── __init__.py
  ├── connector.py      # 统一接口
  ├── mysql_adapter.py  # MySQL适配器
  └── postgres_adapter.py # PostgreSQL适配器
```

**统一接口设计**：
```python
class DBConnector(ABC):
    @abstractmethod
    async def get_instance_status(instance_id: str) -> InstanceStatus: ...
    @abstractmethod
    async def get_sessions(instance_id: str, limit: int) -> list[Session]: ...
    @abstractmethod
    async def get_slow_sqls(instance_id: str, limit: int) -> list[SlowSQL]: ...
    @abstractmethod
    async def get_locks(instance_id: str) -> list[LockInfo]: ...
    @abstractmethod
    async def get_replication_status(instance_id: str) -> ReplicationStatus: ...
    @abstractmethod
    async def get_tablespaces(instance_id: str) -> list[Tablespace]: ...
    @abstractmethod
    async def get_deadlocks(instance_id: str) -> list[Deadlock]: ...
    @abstractmethod
    async def get_connection_pool_status(instance_id: str) -> ConnectionPoolStatus: ...
    @abstractmethod
    async def get_db_type() -> str: ...  # "mysql" or "postgres"
```

**MySQL 特有方法**：
- `show_processlist()` → 适配 `information_schema.PROCESSLIST`
- `show_innodb_status()` → 适配 InnoDB 引擎状态
- `show_master_status()` → 适配主从复制状态

**PostgreSQL 特有方法**：
- `pg_stat_activity()` → 适配 `pg_stat_activity`
- `pg_stat_database()` → 适配 `pg_stat_database`
- `pg_stat_user_tables()` → 适配表统计
- `pg_locks()` → 适配锁视图
- `pg_stat_replication()` → 适配复制状态
- `pg_database_size()` → 适配大小查询

**ConnectionPoolAnalysisTool**：
- MySQL：`SHOW STATUS LIKE 'Threads_connected'` / `SHOW VARIABLES LIKE 'max_connections'`
- PG：`SELECT count(*) FROM pg_stat_activity` / `SHOW max_connections`

**LongTransactionDetectionTool**（PG 特色）：
- `SELECT * FROM pg_stat_activity WHERE state != 'idle' AND query_start < NOW() - INTERVAL '10 minutes'`

### Round 15（R4）- 智能告警增强

#### 3.4.1 新增 AlertAgent

**文件**：`src/agents/alert.py`

**核心能力**：

1. **告警压缩与去重**：
   - 同一根因的多条告警合并为一条

   - 触发条件：预测准确率 > 70%

4. **修复建议生成**：
   - 根据根因生成可执行的操作建议
   - 包含风险评估（是否需要审批）
   - 提供回滚方案

## 四、技术方案建议

### 4.1 数据库适配层核心设计

**统一数据模型**（`src/db/models.py`）：
```python
@dataclass
class InstanceStatus:
    instance_id: str
    db_type: str  # "mysql" | "postgres"
```
