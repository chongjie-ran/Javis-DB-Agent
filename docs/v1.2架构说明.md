### 4.2 资源依赖传播引擎

#### 4.2.1 依赖图结构

```python
class DependencyGraph:
    """
    有向图：边表示"被依赖 → 依赖" 关系
    
    示例边:
        os.hw.cpu → db.da.session  (CPU 影响数据库会话)
        os.hw.memory → db.ds.cache (内存影响缓存)
    """
    
    def __init__(self):
        # 邻接表: resource_id -> [下游资源ID列表]
        self.downstream: dict[str, list[str]] = defaultdict(list)
        # 反向邻接表: resource_id -> [上游资源ID列表]
        self.upstream: dict[str, list[str]] = defaultdict(list)
    
    def add_dependency(self, depended: str, dependent: str, weight: float = 1.0):
        """添加依赖边：depended → dependent（depended 被 dependent 依赖）"""
        self.downstream[depended].append((dependent, weight))
        self.upstream[dependent].append((depended, weight))
```

#### 4.2.2 告警传播算法

```python
def propagate_alert(
    self,
    alert: Alert,
    max_depth: int = 3
) -> list[PropagatedAlert]:
    """
    告警传播：从根因资源向影响到的下游资源传播告警
    
    算法：
    1. 根据告警找到触发资源的 resource_type_id
    2. BFS 遍历依赖图，找到下游资源
    3. 对每层下游资源计算传播置信度
    4. 生成 PropagatedAlert
    """
    propagated = []
    visited = set()
    queue = deque([(alert.resource_type_id, 0, 1.0)])  # (resource_id, depth, confidence)
    
    while queue:
        current_id, depth, confidence = queue.popleft()
        
        if depth > max_depth or current_id in visited:
            continue
        visited.add(current_id)
        
        # 生成传播告警
        if depth > 0:  # 排除原始告警
            propagated.append(PropagatedAlert(
                original_alert=alert,
                propagated_to=current_id,
                propagation_depth=depth,
                confidence=confidence * DEPENDENCY_WEIGHTS.get(current_id, 0.8)
            ))
        
        # 继续传播到下游
        for downstream_id, weight in self.downstream.get(current_id, []):
            if downstream_id not in visited:
                queue.append((
                    downstream_id,
                    depth + 1,
                    confidence * weight
                ))
    
    return propagated
```

#### 4.2.3 根因分析算法

```python
def find_root_cause(
    self,
    symptom_alert: Alert
) -> RootCauseAnalysis:
    """
    逆向根因分析：从症状告警向上游追溯可能的根因
    
    算法：
    1. 从症状资源反向遍历依赖图
    2. 收集所有上游资源和传播路径
    3. 结合知识库匹配可能根因
    4. 输出带置信度的根因列表
    """
    upstream_resources = []
    queue = deque([(symptom_alert.resource_type_id, [symptom_alert.resource_type_id])])
    
    while queue:
        current_id, path = queue.popleft()
        
        for upstream_id, weight in self.upstream.get(current_id, []):
            if upstream_id not in path:  # 避免环
                new_path = path + [upstream_id]
                upstream_resources.append({
                    "resource_id": upstream_id,
                    "chain": new_path,
                    "weight": weight,
                    "depth": len(new_path) - 1
                })
                queue.append((upstream_id, new_path))
    
    # 按深度和权重排序
    upstream_resources.sort(key=lambda x: (x["depth"], x["weight"]), reverse=True)
    
    return RootCauseAnalysis(
        possible_causes=upstream_resources[:5],
        recommended_checks=self._get_checks_for_resources(upstream_resources),
        confidence=self._calculate_confidence(upstream_resources)
    )
```

### 4.3 向量检索架构

#### 4.3.1 向量生成流程

```python
class EmbeddingService:
    """向量嵌入服务"""
    
    def __init__(self):
        self.ollama = OllamaClient()
        self.model = "nomic-embed-text"  # 高质量开源嵌入模型
    
    def embed(self, text: str) -> list[float]:
        """生成 768 维向量"""
        response = self.ollama.embeddings(
            model=self.model,
            prompt=text
        )
        return response["embedding"]
    
    def embed_knowledge_entry(self, entry: KnowledgeEntry) -> list[float]:
        """为知识条目生成联合文本"""
        # 拼接关键字段生成表示文本
        composite = f"""
        类型: {entry.type}
        名称: {entry.name}
        描述: {entry.description}
        症状: {', '.join(entry.symptoms) if entry.symptoms else ''}
        原因: {', '.join(entry.possible_causes) if entry.possible_causes else ''}
        处置: {entry.resolution if entry.resolution else ''}
        """
        return self.embed(composite.strip())
```

```python
    def should_trigger(self, trigger_type: str, context: dict) -> bool:
        """判断是否应该触发"""
        if trigger_type == "on_new_alert":
            count = context.get("alert_count", 0)
            matched = context.get("rule_matched", False)
            return count >= 3 and not matched
        return True
```

#### 4.4.3 自演化安全机制

```python
class EvolutionSafety:
    """自演化安全机制"""
    
    # 高风险操作黑名单
    BLACKLIST = [
        "DROP DATABASE",
        "TRUNCATE TABLE",
        "DELETE FROM users",
        "ALTER SYSTEM",  # 危险配置操作
    ]
    
    def validate_generated_rule(self, rule: AlertRule) -> ValidationResult:
        """验证生成的告警规则"""
        
        # 1. 检查危险操作
        for keyword in self.BLACKLIST:
            if keyword in str(rule.check_steps) or keyword in str(rule.resolution):
                return ValidationResult(
                    valid=False,
                    error=f"检测到危险操作: {keyword}"
                )
        
        # 2. 检查风险级别
        if rule.risk_level in ["L4", "L5"]:
            return ValidationResult(
                valid=True,
                requires_approval=True,  # 需人工审批
                approval_note="高风险规则需人工确认"
            )
        
        return ValidationResult(valid=True)
```

## 5. 数据流设计

### 5.1 告警诊断完整流程（v1.2）

```
1. 告警事件 → Gateway
   │
   ├─→ AlertCorrelator（告警关联）
   │   ├─→ 匹配已有告警规则
   │   └─→ 查找关联告警
   │
   ├─→ DependencyPropagator（依赖传播）⭐ v1.2
   │   ├─→ 传播告警到下游资源
   │   └─→ 逆向追溯根因
   │
   └─→ KnowledgeBaseService（知识检索）
       ├─→ 匹配告警规则（向量+关键词混合检索）
       ├─→ 查找相关 SOP
       └─→ 查找相似案例
           │
           ↓
   诊断Agent 接收上下文：
   - 原始告警 + 关联告警
   - 传播路径（DependencyPropagator 输出）
   - 匹配的知识条目（告警规则/SOP/案例）
   - 观察点元数据（ObservationPointService）⭐ v1.2
           │
           ↓
   诊断Agent 执行诊断：
   1. 根据告警类型查询观察点元数据
   2. 结合资源分类判断影响范围
   3. 参考传播链推断根因
   4. 生成诊断报告（含根因+置信度+处置建议）
           │
           ↓
   输出：
   - 根因分析结果
   - 置信度评分
   - 传播路径可视化
   - 处置建议（含关联 SOP）
   - 观察点元数据（指标描述、阈值）
```

### 5.2 知识自演化流程

```
定时任务（每日凌晨）或 事件触发（新告警无匹配）

Step 1: 数据采集
   ObservableDataCollector.collect(time_range)
   → alerts, metrics, logs

Step 2: 覆盖度评估
   CoverageEvaluator.evaluate(alerts)
   → CoverageReport(uncovered_analysis)

Step 3: 知识生成
   if uncovered_analysis.needs_generation:
       KnowledgeGenerator.generate(uncovered)
       → GeneratedKnowledge

Step 4: 知识验证
   KnowledgeValidator.validate(generated)
   → ValidationResult

Step 5a: 低风险 → 自动入库
   KnowledgeBaseService.create(knowledge)
   → 自动更新数据库 + 重建向量索引

Step 5b: 高风险 → 人工审批
   生成 UpdateProposal
   → 等待管理员审批
   → 审批通过后入库
```

## 6. 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 知识库存储 | SQLite | 轻量、无部署依赖、支持 JSON |
| 向量数据库 | Chroma | 轻量、支持本地部署、与 LLM 生态集成 |
| 向量嵌入模型 | nomic-embed-text | 开源、高质量、768 维 |
| LLM 模型 | Qwen3.5 + Ollama | 已有基础设施 |
| 依赖图 | NetworkX | Python 原生、图算法丰富 |
| 定时任务 | APScheduler | 轻量、支持 cron 表达式 |

## 7. 性能考虑

| 场景 | 目标 | 方案 |
|------|------|------|
| 知识检索 | < 100ms | 向量索引缓存 + 混合检索 |
| 告警传播 | < 50ms | 依赖图预加载 + BFS 剪枝 |
| 自演化生成 | < 30s | LLM 调用超时控制 + 增量生成 |
| 向量索引重建 | < 5min（万级条目） | 增量更新 + 后台任务 |
