# Javis-DB-Agent v1.2 需求说明

> 版本：v1.2 | 日期：2026-03-29 | 状态：初稿 | 制定人：道衍（架构师）

---

## 1. 版本背景与目标

### 1.1 背景

v1.0/v1.1 构建了基础的 Multi-Agent 诊断框架，能力包括：
- 多 Agent 协同（编排、诊断、风险、SQL分析、巡检）
- 告警关联推理链（A→B→C）
- 知识库（告警规则、SOP、案例）
- MySQL/PostgreSQL 双引擎适配

### 1.2 v1.2 核心目标

**聚焦可观测性建模能力增强**，解决当前架构的关键不足：

| 不足 | 现状 | v1.2 目标 |
|------|------|-----------|
| 知识库无分类体系 | 告警规则平铺，缺少实体/资源维度 | 引入"实体-资源-观察点"三维分类体系 |
| 告警关联无传播机制 | 仅依赖预定义规则，无依赖传播 | 支持跨资源依赖传播，增强根因分析 |
| 告警可解释性弱 | 指标→告警缺乏元数据描述 | 定义关键指标"观察点元数据" |
| 知识库管理原始 | YAML 文件手工维护 | 迁移至结构化数据库，支持向量检索 |
| 知识库无法自演化 | 人工更新，无自动化 | LLM+可观测驱动知识库自演化闭环 |

## 2. 能力提升项

### P1 - 能力提升项（必须实现）

#### P1.1 引入"实体-资源"分类体系到知识库

**目标**：将 benchmark 的分类体系引入知识库，提升诊断精度。

**分类体系**：

```
实体层
├── 操作系统
│   ├── 资源大类：硬件资源、软件资源
│   │   ├── 硬件资源小类：CPU、内存、存储IO、文件系统、网络IO
│   │   └── 软件资源小类：进程、文件、目录、系统服务
│   └── 支持的实体小类：Linux、Windows
│
└── 数据库
    ├── 资源大类：数据对象、数据存储、数据访问、系统运行后台、系统灾备
    │   ├── 数据对象小类：系统数据、业务数据
    │   ├── 数据存储小类：缓存、永久存储
    │   ├── 数据访问小类：会话、事务、数据操作指令
    │   ├── 系统运行后台小类：数据对象管理、数据存储管理
    │   └── 系统灾备小类：数据库容灾、数据库备份
    └── 支持的实体小类：MySQL、PostgreSQL、OpenGauss、Oracle、达梦
```

**需求**：
- 知识库条目（告警规则、SOP、案例）需标注关联的"实体-资源"分类
- 诊断时可根据告警的实体/资源类型检索匹配知识
- 支持多层级分类（实体→资源大类→资源小类）

**验收标准**：
- 告警规则关联到实体-资源分类覆盖率 ≥ 90%
- SOP 关联到实体-资源分类覆盖率 ≥ 80%
- 诊断检索时支持按实体/资源过滤

#### P1.2 扩展告警关联支持资源依赖传播

**目标**：基于资源依赖关系，实现告警的跨资源传播，增强根因分析。

**依赖关系**（来自 benchmark）：

| 被依赖资源 | 依赖方资源 | 传播示例 |
|-----------|-----------|---------|
| 硬件资源（CPU/内存/存储IO） | 所有软件/数据库资源 | CPU高 → 数据库响应慢 |
| 软件资源（进程） | 数据库进程 | 进程OOM → DB不可用 |
| 数据对象资源 | 数据访问资源 | 统计信息过期 → SQL慢 |
| 数据存储资源（缓存） | 数据对象资源 | 缓存命中率低 → 查询慢 |
| 系统运行后台资源 | 数据对象/存储资源 | VACUUM停滞 → 表膨胀 |

**需求**：
- 告警关联引擎扩展，支持"上游资源异常→下游资源告警"的传播推理
- 资源依赖图支持增删改查
- 根因分析输出包含完整的依赖传播链

**验收标准**：
- 支持至少 3 层依赖传播（如 CPU→数据库→业务）
- 依赖传播路径可追溯、可解释
- 单元测试覆盖主要依赖传播场景

### P2 - 能力提升项（计划实现）

#### P2.1 定义关键指标"观察点元数据"

**目标**：为关键指标定义完整的元数据，提升告警可解释性。

**观察点分类**（来自 benchmark）：

| 大类 | 小类 | 示例 |
|------|------|------|
| 负载类 | 资源使用率、资源使用量、资源吞吐量、负载均衡度 | CPU使用率80%、内存使用量8GB |
| 性能类 | 资源响应时长、资源等待队列长度 | 锁等待时长30s、连接等待队列10 |
| 错误类 | 资源可用性、错误发生频率、资源缺陷 | 磁盘不可用、错误率0.1%、缺陷数3 |
| 安全类 | 资源越权使用情况 | 越权访问次数5 |
| 配置类 | 性能类配置、安全类配置、稳定类配置 | max_connections=1000 |

**需求**：
- 定义观察点元数据 Schema：
  ```yaml
  observation_point:
    id: "DB_CPU_USAGE_RATE"
    name: "CPU使用率"
    entity: "数据库"
    resource_type: "硬件资源/CPU"
    metric_name: "cpu_usage_percent"
    unit: "percent"  # percent/count/ms/gb
    description: "数据库进程的CPU使用率"
    thresholds:
      warning: 70
      critical: 85
    related_alerts:
      - "CPU_HIGH"
      - "CPU_CORE_UTILIZATION"
    data_source: "pg_stat_activity / performance_schema"
  ```
- 知识库条目关联相关观察点
- 告警输出时附带观察点元数据，增强可解释性

**验收标准**：
- 覆盖核心指标（CPU、内存、连接数、QPS/TPS、响应时间）观察点元数据
- 告警响应包含观察点元数据
- 可通过 API 查询观察点元数据

#### P2.2 知识库 YAML→结构化数据库

**目标**：知识管理现代化，从手工 YAML 迁移至结构化数据库。

**现状**：
```
knowledge/
├── alert_rules.yaml  # 告警规则
├── sop/             # SOP 文件（Markdown）
└── cases/           # 案例文件（Markdown）
```

**目标架构**：
```
knowledge/
├── knowledge.db      # SQLite 数据库
│   ├── alert_rules   # 告警规则表
│   ├── sops          # SOP 表
│   ├── cases         # 案例表
│   ├── entity_taxonomy # 实体分类表
│   ├── resource_taxonomy # 资源分类表
│   ├── observation_points # 观察点元数据表
│   ├── resource_dependencies # 资源依赖关系表
│   └── embeddings    # 向量嵌入表（用于语义检索）
└── migrations/       # 数据库迁移脚本
```

**需求**：
- 迁移现有 YAML 数据到 SQLite
- 支持 CRUD API
- 支持向量检索（基于 embedding）
- 向后兼容：可通过脚本导出为 YAML

**验收标准**：
- 现有告警规则、SOP、案例全部迁移
- API 响应时间 < 100ms（单条查询）
- 向量检索 top-k 准确率 > 80%

### P3 - 核心创新（LLM+可观测驱动知识库自演化）

#### P3.1 LLM+可观测驱动知识库自演化架构

**目标**：基于可观测数据，LLM 自动分析知识库覆盖度，自动生成新知识条目。

**核心闭环**：

```
[可观测数据源]
    ↓ 采集
[可观测性数据层]（指标、日志、追踪）
    ↓ 分析
[知识库覆盖度评估]（LLM分析当前知识库盲区）
    ↓ 生成
[新知识条目]（规则/SOP/案例/观察点）
    ↓ 验证
[知识库自动更新]
    ↓ 反馈
[诊断能力增强]
```

**子模块设计**：

##### 3.1.1 可观测性数据采集层

**数据源**：
- 监控指标：`/api/v1/metrics/`（CPU、内存、连接数、QPS/TPS）
- 告警事件：`/api/v1/alerts/`（历史告警）
- 日志：`/api/v1/logs/`（错误日志、慢查询日志）
- 追踪：`/api/v1/traces/`（请求链路）

**采集接口**：
```python
class ObservableDataCollector:
    async def collect_metrics(instance_id: str, time_range: TimeRange) -> list[Metric]
    async def collect_alerts(instance_id: str, time_range: TimeRange) -> list[Alert]
    async def collect_logs(instance_id: str, time_range: TimeRange, level: str) -> list[LogEntry]
    async def collect_traces(instance_id: str, time_range: TimeRange) -> list[Trace]
```

##### 3.1.2 知识库覆盖度评估器

**LLM Prompt**：
```
你是知识库评估专家。给定以下可观测数据：
- 近期的告警事件
- 异常指标模式
- 错误日志

分析当前知识库的覆盖度：
1. 列出已有知识覆盖的告警类型
2. 列出未覆盖的告警类型（无对应规则/SOP）
3. 对未覆盖的告警，分析其可能的根因和处置方式
4. 评估是否需要生成新的告警规则或SOP

输出格式：
{
  "covered": ["ALERT_TYPE_1", "ALERT_TYPE_2"],
  "uncovered": [
    {
      "alert_type": "NEW_ALERT_X",
      "description": "描述",
      "estimated_root_causes": ["原因1", "原因2"],
      "suggested_resolution": "建议处置"
    }
  ],
  "knowledge_gaps": ["gap1", "gap2"]
}
```

**触发条件**：
- 定时触发：每日凌晨分析前一天的告警数据
- 事件触发：新告警类型出现（无匹配规则）时触发

##### 3.1.3 知识条目生成器

**LLM Prompt（告警规则生成）**：
```
基于以下可观测数据，生成告警规则：
- 告警类型：{alert_type}
- 告警描述：{description}
- 相关指标：{related_metrics}
- 历史案例：{similar_cases}

生成标准格式的告警规则 YAML，包含：
- alert_type（告警类型编码）
- name（告警名称）
- description（描述）
- severity（严重程度）
- risk_level（风险级别 L1-L5）
- symptoms（症状）
- possible_causes（可能原因）
- check_steps（检查步骤）
- resolution（处置方式）
```

**LLM Prompt（SOP 生成）**：
```
基于以下案例，生成标准操作程序（SOP）：

案例信息：
- 问题描述：{problem}
- 诊断过程：{diagnosis}
- 处置措施：{resolution}
- 耗时：{duration}

生成标准 SOP，包含：
- 标题
- 适用范围
- 前置条件
- 操作步骤（详细）
- 验证方法
- 回滚方案
- 注意事项
```

##### 3.1.4 知识验证与更新

**验证机制**：
- LLM 自检：生成的知识条目是否符合 Schema
- 重复检测：新生成的规则是否与现有规则重复（向量相似度 > 0.9 则认为重复）
- 人工确认：高风险（L4/L5）知识条目需人工确认后入库

**更新流程**：
```
生成 → 自检 → 重复检测
  ↓
低风险(L1-L3) → 自动入库 → 反馈记录
  ↓
高风险(L4-L5) → 人工确认 → 确认后入库
```

## 3. 数据模型

### 3.1 实体-资源分类模型

```sql
-- 实体表
CREATE TABLE entities (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    type TEXT NOT NULL,  -- 'os' / 'db'
    parent_id TEXT,
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (parent_id) REFERENCES entities(id)
);

-- 资源分类表
CREATE TABLE resource_types (
    id TEXT PRIMARY KEY,
    entity_id TEXT NOT NULL,
    category TEXT NOT NULL,  -- 'hardware'/'software'/'data_object'/...
    name TEXT NOT NULL,
    description TEXT,
    FOREIGN KEY (entity_id) REFERENCES entities(id)
);

-- 观察点元数据表
CREATE TABLE observation_points (
    id TEXT PRIMARY KEY,
    resource_type_id TEXT NOT NULL,
    name TEXT NOT NULL,
    metric_name TEXT NOT NULL,
    unit TEXT NOT NULL,  -- 'percent'/'count'/'ms'/'gb'
    description TEXT,
    threshold_warning REAL,
    threshold_critical REAL,
    related_alerts TEXT,  -- JSON array
    data_source TEXT,
    FOREIGN KEY (resource_type_id) REFERENCES resource_types(id)
);

-- 资源依赖关系表
CREATE TABLE resource_dependencies (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    depended_resource_id TEXT NOT NULL,  -- 被依赖资源
    dependent_resource_id TEXT NOT NULL, -- 依赖方资源
    dependency_type TEXT,  -- 'affects'/'requires'/'contains'
    propagation_weight REAL DEFAULT 1.0,  -- 传播权重
    description TEXT,
    FOREIGN KEY (depended_resource_id) REFERENCES resource_types(id),
    FOREIGN KEY (dependent_resource_id) REFERENCES resource_types(id)
);
```

### 3.2 知识库模型

```sql
-- 告警规则表
CREATE TABLE alert_rules (
    id TEXT PRIMARY KEY,
    alert_type TEXT NOT NULL UNIQUE,
    name TEXT NOT NULL,
    description TEXT,
    severity TEXT NOT NULL,  -- 'info'/'warning'/'critical'
    risk_level TEXT NOT NULL,  -- 'L1'-'L5'
    symptoms TEXT,  -- JSON array
    possible_causes TEXT,  -- JSON array
    check_steps TEXT,  -- JSON array
    resolution TEXT,
    entity_id TEXT,  -- 关联实体
    resource_type_id TEXT,  -- 关联资源类型
    observation_point_ids TEXT,  -- JSON array
    enabled BOOLEAN DEFAULT 1,
    version INTEGER DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (entity_id) REFERENCES entities(id),
    FOREIGN KEY (resource_type_id) REFERENCES resource_types(id)
);

-- SOP表
CREATE TABLE sops (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    applicable_entities TEXT,  -- JSON array
    applicable_alert_types TEXT,  -- JSON array
    risk_level TEXT,
    estimated_duration TEXT,
    version INTEGER DEFAULT 1,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 案例表
CREATE TABLE cases (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    problem TEXT NOT NULL,
    diagnosis TEXT NOT NULL,
    resolution TEXT NOT NULL,
    duration TEXT,
    entity_id TEXT,
    resource_type_id TEXT,
    related_alert_types TEXT,  -- JSON array
    lessons_learned TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (entity_id) REFERENCES entities(id),
    FOREIGN KEY (resource_type_id) REFERENCES resource_types(id)
);

-- 向量嵌入表
CREATE TABLE embeddings (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_type TEXT NOT NULL,  -- 'alert_rule'/'sop'/'case'
    entity_id TEXT NOT NULL,
    embedding BLOB NOT NULL,  -- 二进制向量
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(entity_type, entity_id)
);
```

## 4. 验收标准汇总

| 优先级 | 功能 | 验收标准 |
|--------|------|----------|
| P1 | 实体-资源分类体系 | 告警规则覆盖率 ≥ 90%，SOP覆盖率 ≥ 80% |
| P1 | 资源依赖传播 | 支持 ≥ 3 层依赖传播，路径可追溯 |
| P2 | 观察点元数据 | 核心指标全覆盖，告警响应含元数据 |
| P2 | 知识库结构化 | 全量迁移，向量检索可用，API 响应 < 100ms |
| P3 | 知识自演化 | 覆盖度评估可执行，自动生成知识条目 |

## 5. 风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 生成知识的质量不可控 | 错误知识导致误诊 | 高风险知识需人工确认 |
| 向量检索准确率不足 | 检索不到相关知识 | 先用关键词+向量混合检索 |
| 依赖图数据不完整 | 传播路径遗漏 | 先构建核心依赖，预留扩展接口 |
