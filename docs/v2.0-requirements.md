## 1. 概述

### 1.1 项目背景

Javis DB Agent 是面向 zCloud 数据库运维平台的智能体系统，当前已完成 V1.x 阶段建设，实现了多 Agent 协同编排、告警关联推理、意图识别路由等核心能力。根据 V1.1 业务架构与 V1.1 技术架构文档的差距分析，系统在安全治理、知识层、感知层存在显著缺口，制约产品化与阶段二/三演进。

### 1.2 V2.0 目标定位

V2.0 是从"Copilot 辅助"走向"受控闭环"的关键版本，核心目标是：

- **P0-1**：建立 SQL AST 护栏 + 执行回流闭环，实现从"建议输出"到"受控执行"
- **P0-2**：构建知识图谱 + 案例库，实现结构化知识推理与 RAG 混合检索增强
- **P0-3**：建立拓扑/配置/变更感知能力，打通真实 zCloud API 对接

### 1.3 现状差距总结（基于 V1.1 架构文档）

| 层次 | 架构要求 | Javis 现状 | 优先级 |
|------|---------|-----------|--------|
| 安全治理层 | SQL AST 护栏 + 执行回流验证 + SOP 执行器 | 仅参数级校验，无 AST，无回流闭环 | **P0** |
| 知识层 | 知识图谱 + 50 案例 + RAG 混合检索 | 案例 3 个，无图谱，RAG 基础 | **P0** |
| 感知层 | 拓扑/配置/变更感知 + 真实 API | 全 Mock 数据，无拓扑/配置/变更感知 | **P0** |
| 缺失 Agent | 变更管理/配置风险/HA 诊断/处置决策 | 4 类 Agent 全部缺失 | P1 |
| 阶段演进 | 阶段二闭环 + 阶段三自治 | 阶段一基本完成，阶段二部分实现，阶段三未开始 | P1/P2 |

## 2. P0-1 安全治理层增强（SQL 护栏 + 执行闭环）

### 2.1 需求详述

#### 2.1.1 业务价值

| 价值维度 | 具体说明 |
|---------|---------|
| **产品化门槛** | 当前 SQL 护栏仅靠参数校验，无 AST 校验，无法满足金融/医疗客户的安全要求 |
| **核心差异点** | 执行闭环是区别"问答助手"与"智能体"的分水岭，没有闭环就不是真正的智能体 |
| **用户信任度** | 有 AST 校验 + 执行回流验证，用户才敢让 Agent 真正执行动作 |
| **阶段演进基础** | 执行闭环是阶段二（受控闭环）和阶段三（部分自治）的前置条件 |

#### 2.1.2 技术可行性

| 技术点 | 可行性评估 | 现有基础 |
|--------|-----------|---------|
| SQL AST 解析 | ✅ 可行 | Python `sqlglot`（支持 20+ 数据库方言）可直用 |
| SOP 执行器 | ✅ 可行 | 现有 `knowledge/sop/` 目录，10 个 SOP 已结构化 |
| 执行回流验证 | ✅ 可行 | 现有 `gateway/` 中 post_execute 钩子可扩展 |
| Pre/Post Check | ✅ 可行 | 现有 `tools/base.py` 的 Tool 基类可扩展 |

#### 2.1.3 依赖关系

```
P0-1 安全治理层
├── 依赖：PolicyEngine（src/gateway/policy_engine.py）
├── 依赖：ApprovalGate（src/models/approval.py）
├── 依赖：Audit 链（src/gateway/audit.py）
├── 新增：src/security/sql_guard/    SQL 护栏模块
├── 新增：src/security/execution/   执行闭环模块
└── 增强：tools/base.py Pre/Post Check 机制
```

#### 2.1.4 风险点

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| AST 解析误杀正常 SQL | 中 | 白名单机制 + 模板化 SQL 优先 |
| SOP 执行器状态机复杂度 | 高 | 分阶段交付，先线性 SOP 再分支 |
| 执行回流超时/死循环 | 高 | 3 次 max retry + 超时中断 + 人工升级 |
| SOP 步骤涉及危险动作 | 中 | 所有 SOP 步骤必须通过 PolicyEngine 风险分级 |

#### 2.2.2 数据流设计

```
用户请求（执行慢 SQL 优化 SOP）
        │
        ▼
┌────────────────────────┐
│  Orchestrator          │ 接收意图，选择 SOP 执行路径
└────────────────────────┘
        │
        ▼
┌────────────────────────┐
│  SOPExecutor           │ 解析 SOP 步骤序列，生成执行计划
│  （状态机）             │
└────────────────────────┘
        │ 逐步骤执行
        ▼
┌────────────────────────┐
│  SQLGuard              │ AST 解析 → 白名单匹配 → 风险评级
│  （护栏）               │ 输出：ALLOWED / DENIED / NEED_APPROVAL
└────────────────────────┘
        │
        ├─── DENIED ───→ 终止，返回拒绝原因
        │
        ├─── NEED_APPROVAL ──→ 触发 ApprovalGate 双人审批
        │                              │
        │                         审批通过 ──→ 继续
        │                         审批拒绝 ──→ 终止
        │
        └─── ALLOWED ──→ PolicyEngine 二次检查
                              │
                              ▼
                     ┌────────────────────────┐
                     │  PreCheck Hook          │ 执行前检查（验证实例状态）
                     └────────────────────────┘
                              │
                              ▼
                     ┌────────────────────────┐
                     │  ToolExecutor           │ 调用受控工具
                     └────────────────────────┘
                              │
                              ▼
                     ┌────────────────────────┐
                     │  PostCheck Hook         │ 执行后验证 + Audit 写入
                     │  + Feedback            │
                     └────────────────────────┘
                              │
                         状态未改善 ──→ 下一轮 SOP（最多 3 轮）
                         状态改善 ──→ 执行完成
```

#### 2.2.3 现有系统集成

| 现有模块 | 集成方式 |
|---------|---------|
| `PolicyEngine` | SQLGuard 复用风险分级，ALLOWED 结论仍需 PolicyEngine 二次确认 |
| `ApprovalGate` | L4/L5 高风险动作复用现有双人审批 |
| `Audit` | 执行闭环全流程写入 Audit 哈希链，含 SOP 步骤级审计 |
| `ToolRegistry` | SOPExecutor 作为特殊 Tool 注册 |
| `tools/base.py` | PreCheck/PostCheck 作为 Tool 基类扩展钩子 |

### 2.3 接口定义

#### 2.3.1 SQLGuard 接口

```python
# src/security/sql_guard/guard.py

class SQLGuardStatus(Enum):
    ALLOWED = "allowed"
    DENIED = "denied"
    NEED_APPROVAL = "need_approval"
    TEMPLATE_MATCHED = "template_matched"

@dataclass
class SQLGuardResult:
    status: SQLGuardStatus
    reason: str
    risk_level: RiskLevel
    matched_template: Optional[str] = None
    ast_tree: Optional[Any] = None

class SQLGuard:
    def guard(self, sql: str, context: PolicyContext) -> SQLGuardResult:
        """统一入口：SQL → 校验结果"""

    def register_template(self, name: str, sql_template: str, risk_level: RiskLevel):
        """注册白名单 SQL 模板"""

    def validate_ast(self, sql: str) -> ValidationResult:
        """AST 语义校验：禁止 TRUNCATE/DROP DATABASE 等危险操作"""

    def check_dml_bounds(self, sql: str) -> bool:
        """DML 边界检查：WHERE 必填、强制 LIMIT、禁止无限制 DELETE"""
```

#### 2.3.2 SOPExecutor 接口

```python
# src/security/execution/sop_executor.py

class SOPStatus(Enum):
    PENDING = "pending"
    RUNNING = "running"
    WAITING_APPROVAL = "waiting_approval"
    COMPLETED = "completed"
    FAILED = "failed"
    ROLLED_BACK = "rolled_back"

@dataclass
class SOPStepResult:
    step_id: str
    status: SOPStatus
    tool_name: str
    input_params: dict
    output: Any
    pre_check: CheckResult
    post_check: CheckResult
    error: Optional[str] = None

class SOPExecutor:
    def load_sop(self, sop_id: str) -> SOPDefinition:
        """加载 SOP 定义（从 knowledge/sop/ 读取 YAML）"""

    def plan(self, sop_id: str, context: dict) -> SOPPlan:
        """生成执行计划：拓扑排序 + 依赖分析"""

    def execute(self, sop_id: str, context: dict) -> SOPExecutionResult:
        """执行 SOP，支持暂停/恢复/中止"""

    def rollback(self, execution_id: str) -> RollbackResult:
        """回退：按逆序执行回退步骤"""
```

## 4. P0-3 感知层增强（拓扑/配置/真实 API）

### 4.1 需求详述

#### 4.1.1 业务价值

| 价值维度 | 具体说明 |

| **真实场景测试基础** | Mock 数据无法验证真实场景，感知层增强后才能做真实场景测试 |
| **为阶段三自治铺垫** | 自治的前提是能主动感知环境变化，不能只等用户提问 |
| **ROI 高** | 拓扑/配置感知工具开发完成后，可被多个 Agent 复用 |

#### 4.1.2 技术可行性

| 技术点 | 可行性评估 | 现有基础 |
|--------|-----------|---------|
| zCloud API 对接 | ✅ 可行 | 已有 `src/real_api/` 目录，12 个 Router |
| 拓扑感知工具 | ✅ 可行 | zCloud 平台有实例关系数据，可封装为 Tool |
| 配置感知工具 | ✅ 可行 | `src/real_api/routers/parameters.py` 已存在 |
| 变更记录感知 | ⚠️ 需调研 | 需确认 zCloud 工单系统是否有变更记录 API |
| Mock→Real 切换 | ✅ 可行 | 现有 real_api/ vs mock_api/ 双通道架构可复用 |
