# zCloud 数据库运维智能体业务架构

## 1. 架构定位

zCloud 数据库运维智能体，是构建在 zCloud 现有数据库监控、性能分析、诊断运维与自动化能力之上的一层**智能化运维业务中枢**。

它不是单纯的“大模型问答助手”，也不是传统的规则告警系统，而是一个面向数据库场景的**感知—理解—推理—决策—执行**闭环系统，目标是把 zCloud 从“看见问题、展示问题”的运维平台，升级为“识别问题、理解问题、分析问题、建议处置、自动执行”的数据库智能运维平台。

其业务定位可以概括为：

**以数据库为核心对象，以运行状态理解为中枢，以智能诊断与自动化运维为主线，实现数据库运维从监控告警走向智能闭环。**

---

## 2. 业务目标

zCloud 数据库运维智能体的业务目标，建议定义为 4 个方面。

### 2.1 从“数据展示”走向“状态理解”

传统监控平台主要提供指标、图表、SQL 统计和告警。
智能体要进一步把这些数据转化为：

* 当前处于什么状态
* 哪类问题最突出
* 哪些对象最值得关注
* 当前风险等级如何

### 2.2 从“现象发现”走向“根因分析”

不仅告诉用户：

* CPU 高
* 会话多
* 锁等待严重
* 某 SQL 慢

还要进一步分析：

* 问题主要由什么引起
* 哪些信号是主因，哪些是伴随现象
* 可能的根因链路是什么

### 2.3 从“人工判断”走向“智能建议”

智能体要把数据库专家经验沉淀为机器可执行的分析和建议能力，输出：

* 风险说明
* 优先级排序
* 处置建议
* 推荐动作

### 2.4 从“建议输出”走向“受控自动化运维”

在低风险、可验证、可回滚的场景下，支持自动执行或半自动执行：

* 诊断采集
* 会话处置
* 统计刷新
* 运维检查
* 常规优化动作

## 4. 业务对象

* 主备/复制链路
* 数据库用户与连接

* SQL、执行计划、统计信息
* 锁、会话、事务
* 备份、恢复、归档、HA 状态

## 4.2 运维对象

* 告警事件
* 风险事件
* 诊断任务
* 自动化任务
* 诊断报告
* 处置建议
* 运维动作

## 5. 智能体业务主线

从业务上看，zCloud 数据库运维智能体应围绕两条主线展开。

## 5.1 主线一：数据库智能诊断

这是最核心的主业务线，目标是：

**从现象感知 → 状态识别 → 根因分析 → 诊断建议**

主要覆盖：

* 实例状态诊断
* 性能瓶颈诊断
* SQL 性能诊断
* 锁与事务诊断
* 容量与对象诊断
* 高可用与保护状态检查
* 安全与访问风险识别

## 5.2 主线二：自动化运维执行

这是诊断结果的落地业务线，目标是：

**从诊断建议 → 动作选择 → 受控执行 → 结果反馈**

主要覆盖：

* 诊断采集自动化
* 常规检查自动化
* 低风险处置自动化
* 半自动运维动作
* 运维执行结果回收

## 6. 智能认知闭环业务架构

这是整个业务架构的中枢。

# 6.1 感知层（Perception）

感知层负责回答：

**数据库发生了什么。**

感知层不是单一数据采集，而是 zCloud 现有观测能力与实时查询能力的统一感知入口。

### 主要业务输入

#### 1）zCloud 监控指标

* CPU、内存、IO、网络
* QPS/TPS
* 活跃会话数、连接数
* 缓存命中率
* 表空间与容量
* 复制延迟、日志状态

#### 2）zCloud SQL 性能数据

* Top SQL
* 慢 SQL
* SQL 执行次数、耗时、IO、CPU
* 执行计划历史
* SQL 等待事件分布

#### 3）SQL 子集查询能力

用于按需精细取证：

* 活跃会话
* 锁等待/阻塞链
* 长事务
* 执行计划
* 对象元数据
* 参数与状态视图
* 复制/备份状态检查

#### 4）智能感知能力

* 基线检测
* 异常指标检测
* 指标关联分析
* 模式识别
* 多指标共振识别

### 感知层业务输出

* 原始事实（facts）
* 异常信号（signals）
* 关联信号
* 候选问题线索

---

# 6.2 理解层（State Model）

理解层负责回答：

**这些信号意味着数据库当前处于什么状态。**

理解层是把数据和信号转成状态语义的中间层，是整个智能体的业务核心桥梁。

### 主要业务职责

* 信号归并
* 状态识别
* 状态分级
* 状态摘要
* 状态关系组织

### 核心状态域

#### 1）负载与资源状态

* 高负载
* CPU 瓶颈
* IO 瓶颈
* 内存压力
* 缓存效率低

#### 2）并发与事务状态

* 高并发
* 锁竞争
* 严重阻塞
* 长事务
* 会话风暴
* 连接压力

#### 3）SQL 与执行状态

* 热点 SQL
* 慢 SQL 聚集
* 执行计划退化
* 全表扫描主导
* 索引缺失风险
* SQL 频率突增

#### 4）对象与容量状态

* 容量风险
* 表空间压力
* 对象增长异常
* 统计信息过期
* 临时空间压力
* 对象膨胀风险

#### 5）高可用与数据保护状态

* 复制延迟
* 复制不健康
* 高可用降级
* 备份失败风险
* 恢复可用性不足

### 理解层业务输出

* 当前状态集合
* 主问题类别
* 主瓶颈方向
* 严重程度
* 置信度

---

# 6.3 推理层（Reasoning）

推理层负责回答：

**为什么会这样。**

这一层是智能体“专家化”的关键，核心不是单纯复述状态，而是构造问题链路、根因关系和优先级。

### 主要业务职责

* 多状态关联分析
* 根因候选识别
* 主因/伴随现象分离
* 因果链构建
* 风险影响分析

### 典型推理问题

* 锁竞争是否由长事务引发
* IO 瓶颈是否由热点 SQL 引起
* 慢 SQL 是否由执行计划退化导致
* 复制延迟是否由主库压力放大导致
* 高负载是系统整体问题还是少数 SQL 问题
* 容量问题是否已影响性能与可用性

### 推理层业务输出

* 根因候选
* 证据链
* 影响范围
* 优先级排序

---

# 6.4 决策层（Decision）

决策层负责回答：

**应该做什么。**

这一层将推理结果转化为可执行建议或动作。

### 主要业务职责

* 生成诊断结论
* 生成优化建议
* 给出处置方案
* 选择自动化动作
* 区分建议、半自动、自动执行

### 决策输出类型

#### 1）建议型输出

* SQL 优化建议
* 索引建议
* 参数排查建议
* 容量治理建议
* HA 风险处理建议

#### 2）半自动输出

* 建议执行某检查
* 建议收集更多诊断数据
* 建议 kill 某会话
* 建议刷新统计信息

#### 3）自动化输出

* 自动发起诊断采集
* 自动执行低风险检查
* 自动触发巡检
* 自动生成报告

---

# 6.5 执行层（Action / Execution）

从业务上，执行层负责：

**把决策转成运维动作，并回收结果。**

这里强调的是“受控自动化运维”，不是无限制动作执行。

### 主要执行类型

* 数据采集类动作
* 检查验证类动作
* 低风险优化类动作
* 会话控制类动作
* 运行维护类动作

### 动作原则

* 优先低风险
* 优先可验证
* 优先可审计
* 优先有明确边界
* 高风险动作保留人工确认

### 执行回流

执行结果要回流给感知层，形成闭环：

* 动作是否成功
* 系统状态是否改善
* 是否仍有异常
* 是否需要进入下一轮诊断

## 8. 业务交互入口

从业务使用视角，智能体可通过 4 类入口对外提供能力。

### 8.1 告警入口

用户从告警进入智能体：

* 这个告警意味着什么
* 当前影响有多大
* 可能原因是什么
* 建议先做什么检查

### 8.2 问答入口

用户直接问：

* 当前数据库为什么慢
* 哪些 SQL 最可能有问题
* 当前主瓶颈是 CPU 还是 IO
* 复制延迟的主要原因是什么

### 8.3 诊断入口

用户发起明确诊断任务：

* 慢库诊断
* 锁阻塞诊断
* SQL 性能诊断
* 容量风险诊断
* 主备异常诊断

### 8.4 自动化入口

用户选择自动或半自动运维：

* 自动采集诊断信息
* 自动执行检查
* 自动生成诊断报告
* 半自动执行推荐动作

## 9. 典型业务闭环

### 场景一：性能下降诊断

1. 感知层发现响应时间、active session、IO wait 异常
2. 理解层识别出高负载、IO 瓶颈、热点 SQL
3. 推理层分析出某热点 SQL 全表扫描导致 IO 压力
4. 决策层给出索引优化与 SQL 优化建议
5. 自动化层可执行额外诊断采集
6. 结果回流验证状态是否恢复

### 场景二：锁阻塞诊断

1. 感知层发现 lock wait、blocking chain、长事务
2. 理解层识别出锁竞争、严重阻塞、长事务存在
3. 推理层判断某会话长期未提交导致阻塞扩散
4. 决策层生成处置建议
5. 在受控条件下可发起半自动会话处置

### 场景三：主备异常诊断

1. 感知层发现 replication lag、archive 异常
2. 理解层识别复制延迟、HA 降级
3. 推理层分析可能由主库压力或链路问题引起
4. 决策层输出检查建议与风险提示
5. 自动化层可发起链路检查与补充采集

## 10. 与 zCloud 现有能力的关系

这点在业务架构里必须说清楚。

## 10.1 zCloud 现有能力是智能体的底座

智能体不是替代 zCloud，而是站在 zCloud 上层进行智能化升级。

zCloud 已有能力主要承担：

* 数据采集
* 监控与可观测
* SQL 性能分析
* 告警与巡检
* 诊断工具
* 自动化运维工具
* 经验知识沉淀

## 10.2 智能体新增的是“认知与决策能力”

新增价值主要体现在：

* 信号归并
* 状态识别
* 根因推理
* 建议生成
* 智能交互
* 自动化动作编排与触发

## 10.3 二者关系

可以概括为：

**zCloud 提供感知与执行底座，智能体提供理解、推理与决策中枢。**
