# zCloud 数据库运维智能体（Agent）架构

## 1. 先定义：zCloud 的 Agent 到底是什么

我建议把 zCloud 的 Agent 定义为：

> **面向数据库运维场景的智能决策与执行中枢，通过感知数据库状态、理解运维上下文、调用受控工具能力、执行安全闭环动作，实现从“告警响应”走向“智能诊断、风险预防与自动化处置”的数据库运维智能体体系。**

这里有几个关键词：

* **智能决策**
* **运维场景**
* **受控工具调用**
* **安全闭环**
* **从辅助走向自治**

也就是说，zCloud Agent 不是聊天机器人，也不是 SQL 问答助手，而是 **运维控制平面上的智能体系统**。

---

# 2. zCloud Agent 的目标边界

先把边界定清楚，否则很容易把 Agent 做歪。

## 2.1 它要解决的问题

zCloud Agent 主要解决：

### 一类：认知型问题

* 这次告警真正根因是什么
* 影响范围有多大
* 是数据库问题、主机问题、存储问题，还是应用 SQL 问题
* 当前最值得优先处理的风险是什么

### 二类：决策型问题

* 下一步应该看什么
* 应该做哪些检查
* 应该采用哪种处置路径
* 哪些动作可以自动执行，哪些必须人工审批

### 三类：执行型问题

* 自动做诊断采样
* 自动生成分析报告
* 自动执行低风险 SOP
* 自动联动 zCloud 的运维能力完成闭环

### 四类：知识型问题

* 沉淀 runbook
* 复用历史故障经验
* 把 DBA 经验结构化
* 让系统逐步形成“数据库运维知识中枢”

---

## 2.2 它不应该直接做的事

一开始不建议让 Agent 直接做：

* 任意 SQL 执行
* 任意 DDL/DML 变更
* 任意登录主机执行 shell
* 不受控的数据库参数修改
* 不受控的 kill session / failover / switchover

也就是说：

> **Agent 不能直接拥有“原始破坏能力”，只能通过受控工具平面调用能力。**

这点对 zCloud 非常关键。

---

# 3. zCloud Agent 的总体架构主张

我建议 zCloud 的 Agent 架构主张定义为：

> **“感知层统一汇聚、知识层结构增强、决策层多Agent协同、执行层工具受控、安全层全程护栏”的数据库运维智能体架构。**

这句话可以作为总纲。

对应展开为五层：

1. 感知与上下文层
2. 知识与语义层
3. 智能决策与多 Agent 协同层
4. 工具执行与运维能力层
5. 安全治理与审计层

# 5. 五层架构详细说明

## 5.1 感知与上下文层

这是 Agent 的“眼睛和耳朵”。

zCloud 比通用大模型产品最大的优势，就是你们天然有数据库运维控制面，因此上下文不是靠聊天凑出来的，而是靠平台沉淀出来的。

应接入的数据包括：

### 运行状态

* 实例状态
* CPU / 内存 / IO / 网络
* 会话数
* 锁等待
* 活跃事务
* 主从复制状态
* HA 状态
* 备份状态
* 容量状态

### 性能画像

* Top SQL
* SQL 变更趋势
* 慢 SQL
* 执行计划变化
* 等待事件分布
* 时间段基线偏移

### 运维上下文

* 巡检结果
* 告警事件
* 工单记录
* 变更记录
* 历史故障
* 已执行自动化作业
* 当前值班/责任人

### 拓扑与配置

* 数据库版本
* 参数配置
* 实例角色
* 集群关系
* 应用依赖
* 主机/存储/中间件拓扑

这一层的原则是：

> **Agent 不能只看自然语言输入，必须以 zCloud 平台真实上下文为主。**

---

## 5.2 知识与语义层

这是 Agent 的“长期记忆”。

单靠 LLM 本身做数据库运维是不够的，因为真正有价值的是你们自己的知识资产。

这层建议至少有 5 类知识：

### 1. 运维知识库

* 数据库基础知识
* 各数据库版本特性
* 参数说明
* 常见故障模式

### 2. 告警知识库

* 告警定义
* 触发逻辑
* 常见原因
* 排查路径
* 优先级和影响范围

### 3. SOP / Runbook 库

* 日常巡检 SOP
* 故障响应 SOP
* 性能排查 SOP
* 容量处理 SOP
* 备份异常处理 SOP

### 4. 案例库

* 历史事故
* 根因
* 处置过程
* 结果
* 是否复发
* 哪些动作有效

### 5. 语义模型 / 知识图谱

这是比较关键的一层增强，可以逐步做：

* 实例
* 数据库
* 主机
* 存储
* 参数
* 告警
* SQL
* 业务系统
* 工单
* 责任团队

把这些关系连接起来，Agent 才更容易做“上下文推理”和“影响分析”。

---

## 5.3 智能决策与多 Agent 协同层

这是系统的大脑。

我不建议一开始只做一个“大一统 Agent”，更合适的是 **一个编排 Agent + 多个专业 Agent**。

### 统一编排 Agent

职责：

* 解析用户目标
* 选择调用哪些专业 Agent
* 组织上下文
* 汇总结果
* 决定是否进入执行流程

### 诊断 Agent

职责：

* 对异常事件做根因分析
* 输出诊断路径
* 给出置信度和证据链

### 风险评估 Agent

职责：

* 判断当前问题风险级别
* 判断是否允许自动处置
* 判断是否需要审批

### SQL 分析 Agent

职责：

* 分析 SQL 画像
* 判断慢 SQL、锁阻塞、计划劣化
* 输出优化建议
* 调用 SQL 受控查询工具

### 性能优化 Agent

职责：

* 综合实例、负载、SQL、参数、资源情况
* 判断瓶颈在数据库、主机还是存储
* 给出性能调优方向

### 巡检 Agent

职责：

* 生成巡检结论
* 对日常健康评分
* 输出风险项和治理建议

### 处置决策 Agent

职责：

* 根据风险与 SOP，决定是否执行自动动作
* 或者转人工审批

### 报告生成 Agent

职责：

* 生成 RCA 报告
* 生成巡检报告
* 生成领导摘要
* 生成工单说明

---

## 5.4 工具执行与运维能力层

这是最关键的一层。
**Agent 不直接操作数据库，而是调用受控工具。**

也就是说，zCloud 原有运维能力，要重新封装成 Tool API。

### 工具分类建议

#### 1. 查询类工具

只读、低风险：

* 查询实例状态
* 查询参数
* 查询锁等待
* 查询复制状态
* 查询会话
* 查询慢 SQL
* 查询执行计划
* 查询磁盘与容量

#### 2. 分析类工具

* 诊断报告生成
* Top SQL 归因分析
* 告警关联分析
* 巡检规则执行
* 变更影响分析
* 容量预测分析

#### 3. 执行类工具

严格受控：

* 刷新监控采样
* 触发巡检
* 执行低风险 SOP
* 重跑某类任务
* 重置采集任务
* 刷新元数据
* 执行受控的 kill session
* 执行受控 failover 前检查

#### 4. 联动类工具

* 发通知
* 创建工单
* 变更审批
* 调用自动化平台
* 写入审计记录

这里的核心原则是：

> **大模型永远不直接输出最终命令，而是选择调用哪个工具、传哪些受限参数。**

---

## 5.5 安全治理与审计层

这一层决定这个架构能不能产品化。

### 核心安全机制建议

#### 权限分层

* 查看权限
* 分析权限
* 建议权限
* 执行权限
* 高危执行权限

#### 动作分级

* L1：只读分析
* L2：自动诊断
* L3：低风险自动执行
* L4：需审批执行
* L5：禁止 Agent 自动执行

#### SQL 护栏

* 只允许 SQL 子集
* 只允许查询类模板
* 不允许原始 DDL/DML 直出
* 通过 AST/模板校验
* 强制 limit、超时、只读连接

#### 审批与确认

* 高风险动作必须人工确认
* 支持双人审批
* 支持工单联动

#### 审计留痕

* 谁触发
* Agent 如何推理
* 调用了哪些工具
* 产生了哪些结果
* 是否执行
* 执行结果如何

#### 结果校验

* 动作执行前 pre-check
* 动作执行后 post-check
* 失败后的补偿/回退建议

# 6. zCloud Agent 的三阶段演进路线

这个很重要。不要一开始就说“自治运维”，而要给出渐进路径。

## 阶段一：Copilot 辅助阶段

关键词：**能看、能问、能解释、能建议**

能力：

* 告警解释
* 巡检解读
* 根因初判
* SQL 分析建议
* 报告自动生成

特点：

* 只读为主
* 不自动执行
* 用户决策为主

这是第一阶段最现实的切入点。

---

## 阶段二：受控闭环阶段

关键词：**能决策、能调用工具、能做低风险动作**

能力：

* 自动补充诊断
* 自动执行低风险 SOP
* 自动工单流转
* 自动巡检与问题归类
* 自动生成处置建议并等待确认

特点：

* Tool-based
* 风险分级
* 人机协同

这是 zCloud 真正建立差异的阶段。

---

## 阶段三：自治运维阶段

关键词：**场景自治、策略自治、可审计自治**

能力：

* 某些固定场景自动处理
* 多步诊断 + 多步动作闭环
* 运行结果自动验证
* 形成经验回流

适合先做的场景：

* 采集异常修复
* 低风险巡检整改
* 常见锁阻塞识别与处置建议
* 慢 SQL 诊断闭环
* 主从延迟异常的分层定位
* 容量风险预警与工单化

注意，是“部分场景自治”，不是“全场景自治”。

# 7. zCloud 最适合优先落地的 6 个 Agent 场景

## 7.1 告警解释与根因诊断 Agent

用户问：

* 这个告警是什么意思
* 为什么触发
* 影响范围是什么
* 下一步该查什么

这是最容易起量的场景。

---

## 7.2 慢 SQL / 锁阻塞分析 Agent

用户问：

* 为什么数据库慢了
* 为什么会阻塞
* 哪些 SQL 最可疑
* 是否属于计划劣化

这个场景很贴近数据库运维价值。

---

## 7.3 巡检与健康治理 Agent

自动输出：

* 巡检摘要
* 风险项列表
* 治理优先级
* SOP 建议

这很适合 zCloud 产品表达。

---

## 7.4 参数与配置风险 Agent

识别：

* 配置偏差
* 参数冲突
* 不合理阈值
* 版本兼容风险

这属于高价值低风险。

---

## 7.5 复制 / HA 状态诊断 Agent

分析：

* 主从延迟
* 复制中断
* 日志缺失
* 心跳异常
* 切换风险

这个对金融、医疗、制造都很有意义。

---

## 7.6 自动化运维作业编排 Agent

联动已有自动化能力：

* 巡检
* 报告
* 采集刷新
* 任务重试
* 通知
* 工单

这个场景能把 Agent 和 zCloud 原产品能力真正接起来。

# 8. 一个关键设计原则：LLM 不直接碰数据库内核动作

你前面已经讨论过 SQL 风险问题，所以在 zCloud 里我建议把原则写死：

> **LLM 不直接生成并执行原始数据库操作；所有数据库操作必须经过受控语义工具层与安全策略层。**

也就是：

* 不是 LLM → SQL → DB
* 而是 LLM → Tool Intent → 参数约束 → 安全校验 → Tool Executor → DB

这点非常关键。

# 9. zCloud Agent 的正式版技术架构表述

你可以直接拿下面这段做正式稿开头：

**zCloud 数据库运维智能体采用“感知汇聚、知识增强、多 Agent 协同、工具受控执行、安全审计闭环”的总体架构。平台以 zCloud 原有的多元数据库纳管、监控、巡检、诊断、自动化运维能力为基础，将数据库运行状态、告警事件、SQL画像、历史工单、SOP与案例知识统一汇聚为运维上下文，由统一编排 Agent 调度诊断 Agent、风险评估 Agent、SQL 分析 Agent、巡检 Agent、处置决策 Agent 等专业智能体协同完成问题识别、根因分析、处置建议与受控动作执行。所有执行动作均通过工具化能力接口进行调用，并受权限、审批、风险分级、SQL护栏与审计机制约束，从而实现数据库运维从“看见问题”到“理解问题、建议处置、部分场景自动闭环”的智能化演进。**

# 10. 我建议你下一步继续往下拆的方向

接下来最值得继续细化的是三件事：

2. **Agent 工具层（Tool API）清单设计**
