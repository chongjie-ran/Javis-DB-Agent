## 〇、执行摘要

本报告基于对项目源码的深度静态分析（grep 统计、文件树扫描、关键模块阅读）以及现有文档（CHANGELOG、docs/V2.6-Planning、docs/V2.7-Assessment、docs/round11-architecture-evaluation、docs/进度追踪）综合得出，共发现 **3 个 P0 阻塞项、11 个 P1 高优项、9 个 P2 中优项**。

**最关键的 3 个问题（必须本周修复）**：
1. 🔴 **`data/audit.jsonl` 已膨胀到 8.5 GB（106,714 条记录）且无日志轮转** —— `AuditLogger._load()` 启动时全量加载，导致 `test_list_audit_logs` 单测耗时 8.70s、内存占用 >2GB，存在磁盘爆满风险。
2. 🔴 **敏感数据 `.bak` 文件残留** —— `data/users.bak`、`data/tokens.bak`、`data/auth_secret.key.bak` 包含生产凭证明文副本，存在严重安全/合规风险。
3. 🔴 **三处版本号不一致**（pyproject.toml 1.3.1 / CHANGELOG V3.0.0 / config.py 2.1.0），导致 Round 13/14 任务卡顿 76 天、P1 真显信号缺失 60 天的根因之一。

**主要技术债务**：未使用的依赖（celery/redis）、CORS 宽泛配置、LLM 客户端无重试/熔断、scheduler 包非相对导入、221 处宽泛 `except Exception`、22 处生产代码 `print()`。

### 1.2 已识别的具体 Bug

#### Bug-1: `src/scheduler/__init__.py` 使用非相对导入 🔴 P0
```python
# 当前代码（有 bug）
from task_classifier import TaskType, Task, TaskClassifier
from file_lock import FileLockManager
from task_scheduler import TaskScheduler
```
**问题**：在 `src/scheduler/` 目录内运行测试可工作（因 cwd 在 sys.path），但从项目根目录 `import src.scheduler` 时会因找不到 `task_classifier` 模块而 `ModuleNotFoundError`。

**修复**：
```python
from .task_classifier import TaskType, Task, TaskClassifier
from .file_lock import FileLockManager
from .task_scheduler import TaskScheduler
```

#### Bug-2: 审计日志无轮转机制 🔴 P0
- `data/audit.jsonl` 当前 8.5 GB / 106,714 条
- `AuditLogger._load()` 在 `__init__` 时全量加载到 `self._logs` 列表
- `test_list_audit_logs` 8.70s 耗时 = 启动时加载 8.5GB JSON
- 实际生产中内存占用 >2GB，TTL 清理线程（`PersistentSessionManager._cleanup_thread`）只清理 sessions.db，**不清理 audit.jsonl**

**修复方案**：实现 `RotatingAuditLogger`（按大小 100MB / 按天轮转），启动时仅加载最近 N 条 + 提供链式校验从磁盘流式遍历。

#### Bug-3: `print()` 残留生产代码 🟡 P1
22 处 `print()` 散落在 src/（agent_runner.py、persistent_session.py、orchestrator.py、capacity_agent.py 等），与 `structlog` 体系冲突：
- 无法被 `LOG_LEVEL` 控制
- 输出到 stdout 不会被 JSONRenderer 格式化
- 在 Docker/K8s 环境下难以结构化采集

#### Bug-4: 8 处 `open()` 未用 `with` 🟡 P1
主要在 `src/llm/ollama_client.py`、`src/security/` 模块，可能导致文件描述符泄漏。

### 2.2 可扩展性观察

**已具备的良好设计**：
- Hook 事件驱动（V3.0 新增 8 个 Hook 点）
- 并行 Agent 引擎（`ParallelAgentExecutor`）
- 工具注册表（`tool_registry`）+ MCP 双向同步
- L1-L5 风险分级 + 双人审批（L5）
- 哈希链审计防篡改

**可改进的设计债务**：
1. **意图识别与 Agent 耦合**：`orchestrator.py` 中 18+ 个 `Intent` 枚举 + `INTENT_EXAMPLES` 字面量字典，意图分发逻辑硬编码。**建议**：抽离为 `IntentClassifier` 策略类，支持 LLM 兜底 + 规则前置。
2. **Channel 适配层重复**：`channels/feishu/` 和 `channels/wecom/` 大量重复的消息解析、加解密、签名校验代码。**建议**：抽象 `BaseChannelAdapter` 统一加解密、签名、ACK 流程。
3. **LLM 客户端单点**：`src/llm/ollama_client.py` 仅支持 Ollama，未预留 OpenAI/Anthropic/DeepSeek 接口。**建议**：定义 `LLMProvider` Protocol，新增 `OpenAIProvider` 等实现。
4. **测试间共享 fixtures 不够**：各 round 目录重复定义 `client` fixture、DB setup。**建议**：在 `tests/conftest.py` 集中管理。

## 三、性能优化分析

### 3.1 启动性能

| 启动项 | 当前耗时 | 根因 | 优化方案 |
|--------|----------|------|----------|
| `AuditLogger._load()` | ~3-5s | 全量读 8.5GB JSON | 懒加载 + 仅缓存最新 1000 条 |
| `PersistentSessionManager._load()` | ~0.5s | 全量读 sessions.db | 增量加载 |
| Hook 注册 | <0.1s | ✅ OK | — |
| MCP Server 同步 | ~0.2s | ✅ OK | — |
| 工具注册 | ~0.3s | ✅ OK | — |

### 3.2 运行时性能

**P95 响应时间基线**：749ms（Round 11 测量），远优于 30s SLA。

| 热点 | 现状 | 优化方向 |
|------|------|----------|
| `OllamaClient._get_client()` | 每次调用 `httpx.AsyncClient()` 重建 | 复用 `httpx.AsyncClient` 单例（连接池） |
| `OllamaClient.complete()` | 无 retry/timeout 显式控制 | 加重试 + `Retry-After` 头解析 |
| `api/routes.py` 多处 | `asyncio.wait_for(coro, timeout=30.0)` 硬编码 | 提到 settings 集中管理 |
| `sessions.db` | 未配置 WAL 模式 | 启用 `PRAGMA journal_mode=WAL` |
| `audit.jsonl` 写入 | 每条 log 单独 `open('a')` + `write` | 批量缓冲 + flush（每 100 条或 5s） |

### 3.3 资源使用

| 资源 | 当前 | 风险 |
|------|------|------|
| 磁盘 | audit.jsonl 8.5GB | 🔴 30 天内可能 >20GB |
| 内存 | AuditLogger._logs 持有全部记录 | 🔴 >2GB RAM |
| 文件描述符 | 后台线程多 | 🟡 中等 |
| CPU | Ollama 推理时 100% | ✅ 设计如此 |
