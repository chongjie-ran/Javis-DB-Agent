# 案例：2026-01-15 锁等待超时故障

> 版本：v1.0 | 日期：2026-01-15 | 实例：PROD-ORDER-DB | 状态：已归档

---

## 1. 事件概述

| 项目 | 内容 |
|------|------|
| 时间 | 2026-01-15 14:23:00 |
| 持续时间 | 约15分钟 |
| 影响范围 | 订单系统写入缓慢，部分订单超时 |
| 告警 | ALT-20260115-014 (锁等待超时) |
| 根因 | 长事务未提交，阻塞其他会话 |
| 处置方式 | Kill阻塞会话 |
| 数据丢失 | 无 |

---

## 2. 事件经过

### 14:23 - 收到告警
```
告警：ALT-20260115-014
类型：LOCK_WAIT_TIMEOUT
实例：PROD-ORDER-DB (INS-001)
消息：当前等待时间120秒，阻塞会话数3
```

### 14:24 - 初步分析
```
查询锁等待链：
- 被阻塞会话：SID 1452, 1453, 1454
- 阻塞者会话：SID 1001（持有TX锁）
- 阻塞者状态：ACTIVE，SQL执行中
- 阻塞者持续时间：约8分钟
```

### 14:25 - 定位根因
```
分析SID 1001的SQL：
- SQL：UPDATE orders SET status='PAID' WHERE order_id=12345
- 执行时间：约8分钟
- 原因：该订单关联了大型日志表，单条更新耗时过长
- 该事务未提交，持续持有锁
```

### 14:27 - 决策处置
```
评估：
- 阻塞影响：3个业务会话+订单积压
- Kill风险：低（未提交事务，数据可回滚）
- 业务影响：订单状态需重试处理

决策：Kill阻塞会话 SID 1001
审批：获得DBA负责人同意
```

### 14:28 - 执行处置
```sql
ALTER SYSTEM KILL SESSION '1001,32041' IMMEDIATE;
```

### 14:30 - 验证恢复
```
- 锁等待链清除
- 3个被阻塞会话自动解除
- 订单系统恢复写入
```

### 14:35 - 后续处理
```
1. 联系应用负责人确认订单处理逻辑
2. 发现：订单状态更新未设置超时
3. 要求应用修复：增加SQL执行超时
4. 归档本次事件
```

---

## 3. 根因分析

### 3.1 直接原因
```
长事务未提交，导致锁无法释放
```

### 3.2 深层原因
```
应用层问题：
- 订单状态更新未设置SQL执行超时
- 单次操作涉及过多关联表
- 缺乏事务超时控制

数据库层：
- 未设置idle_session_timeout
- 未启用DDL锁超时
```

### 3.3 改进措施
| 措施 | 负责人 | 完成时间 |
|------|--------|----------|
| 应用增加SQL执行超时 | 应用团队 | 2026-01-20 |
| 设置idle_session_timeout=30min | DBA | 2026-01-16 |
| 建立锁等待监控告警 | DBA | 2026-01-16 |

---

## 4. 经验教训

### 4.1 处置经验
- 锁等待场景，优先分析阻塞链
- Kill会话前必须确认事务是否可回滚
- 同一事务内避免操作多张大表

### 4.2 监控改进
- 增加"长时间持有锁"告警
- 增加idle会话监控
- 关键表增加变更监控

---

## 5. LLM诊断提示

```
本案例中，Agent应该学会：
1. 识别锁等待模式：多条会话同时等待+一条会话持有锁
2. 优先分析阻塞者（blocker），而不是被阻塞者
3. 评估Kill风险：检查事务是否已提交、数据是否关键
4. 关联应用逻辑：识别"设计缺陷导致的锁等待"
```
