# 案例：2026-02-20 慢SQL风暴事件

> 版本：v1.0 | 日期：2026-02-20 | 实例：PROD-REPORT-DB | 状态：已归档

---

## 1. 事件概述

| 项目 | 内容 |
|------|------|
| 时间 | 2026-02-20 09:15:00 - 10:45:00 |
| 持续时间 | 约90分钟 |
| 影响范围 | 报表系统响应缓慢，部分报表超时失败 |
| 告警 | ALT-20260220-008 (慢SQL告警) |
| 根因 | 统计信息过期 + 凌晨批量任务产生大数据变更 |
| 处置方式 | 重新收集统计信息 |
| 数据丢失 | 无 |

---

## 2. 事件经过

### 09:15 - 收到告警
```
告警：ALT-20260220-008
类型：SLOW_QUERY_DETECTED
实例：PROD-REPORT-DB (INS-002)
消息：过去15分钟有127条慢SQL，平均执行时间8.5秒
```

### 09:16 - 初步分析
```
查询Top SQL：
1. SELECT * FROM sales_detail WHERE stat_date = '2026-02-19'
   - 执行次数：45次
   - 平均执行时间：12.3秒
   - 特征：数据量突增（昨天数据量是前天的5倍）

2. SELECT * FROM report_cache WHERE report_id = ?
   - 执行次数：82次
   - 平均执行时间：6.8秒
   - 特征：缓存穿透，每次都全表扫描
```

### 09:20 - 定位根因
```
时间线分析：
- 02:00 凌晨批次开始，插入当天销售数据（500万条）
- 02:30 批次结束，但未收集统计信息
- 09:00 报表查询开始
- 09:15 统计信息仍为昨天数据，优化器选择错误执行计划

确认：统计信息过期导致执行计划不当
```

### 09:30 - 决策处置
```
方案A：立即收集统计信息
- 优点：快速解决
- 缺点：收集过程可能短暂影响性能

方案B：观察等待
- 缺点：持续影响业务

决策：选择方案A，立即收集统计信息
```

### 09:32 - 执行处置
```sql
-- 收集问题表统计信息
BEGIN
  DBMS_STATS.GATHER_TABLE_STATS(
    ownname => 'REPORT_USER',
    tabname => 'SALES_DETAIL',
    cascade => TRUE,
    degree  => 4
  );
END;
/

-- 收集索引统计信息
BEGIN
  DBMS_STATS.GATHER_INDEX_STATS(
    ownname => 'REPORT_USER',
    indname => 'IDX_SALES_DETAIL_DATE',
    degree  => 4
  );
END;
/
```

### 09:40 - 验证恢复
```
- 统计信息收集完成
- 相同SQL重新执行，平均时间降至0.3秒
- 报表系统恢复正常
```

---

## 3. 根因分析

### 3.1 直接原因
```
统计信息过期，优化器生成错误执行计划
```

### 3.2 深层原因
```
1. 批次任务后未触发统计信息收集
2. 监控未覆盖"统计信息过期"场景
3. 缺乏自动统计信息收集机制（AUTO_STATS）
4. 报表SQL未使用绑定变量，每次硬解析
```

### 3.3 改进措施
| 措施 | 负责人 | 完成时间 |
|------|--------|----------|
| 批次任务后自动收集统计信息 | ETL团队 | 2026-02-25 |
| 开启AUTO_STATS收集任务 | DBA | 2026-02-21 |
| 报表SQL改用绑定变量 | 开发团队 | 2026-03-01 |
| 增加统计信息过期监控告警 | DBA | 2026-02-22 |

---

## 4. 经验教训

### 4.1 处置经验
- 大数据变更后必须立即收集统计信息
- 凌晨批次任务应包含统计信息收集步骤
- 慢SQL告警应该关注"突然增多"而非单条SQL

### 4.2 监控改进
- 增加"统计信息陈旧"监控
- 增加"执行计划突变"监控
- 增加"硬解析比例"监控
