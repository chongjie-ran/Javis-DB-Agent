# 主从延迟排查标准流程 (SOP)

## 1. 告警确认

### 1.1 收到告警
- 告警类型: REPLICATION_LAG
- 记录告警时间、实例ID、从库信息
- 检查告警阈值设置

### 1.2 初步判断
```
延迟 < 30秒: 观察，可能自动恢复
延迟 30秒-5分钟: 需要关注，分析原因
延迟 > 5分钟: 紧急处理
```

## 2. 信息收集

### 2.1 检查复制状态
```sql
-- 主库查看binlog位置
SHOW MASTER STATUS;

-- 从库查看复制状态
SHOW SLAVE STATUS\G

-- 关键指标:
--   Slave_IO_Running: Yes/No
--   Slave_SQL_Running: Yes/No
--   Seconds_Behind_Master: 延迟秒数
--   Relay_Log_Pos: 中继日志位置
--   Last_IO_Error: IO线程错误
--   Last_SQL_Error: SQL线程错误
```

### 2.2 分析延迟原因
```sql
-- 查看从库当前执行的SQL
SHOW PROCESSLIST;

-- 查看大事务
SELECT * FROM mysql.slave_execute_by_gtid(
  WHERE event_time > DATE_SUB(NOW(), INTERVAL 1 HOUR)
);

-- 检查网络延迟
ping -c 10 <master_ip>
```

### 2.3 检查从库资源
```bash
# CPU和负载
top -bn1 | head -20

# 磁盘IO
iostat -x 1 5

# 网络带宽
iftop -i eth0
```

## 3. 延迟原因判定

### 3.1 网络问题
**特征:**
- IO线程错误
- ping延迟高
- 带宽占用高

**处理:**
1. 检查网络链路
2. 联系网络团队
3. 考虑跨机房复制优化

### 3.2 从库负载高
**特征:**
- 从库CPU/IO使用率高
- 大量查询在从库执行

**处理:**
1. 优化从库查询
2. 限制从库读操作
3. 增加从库资源

### 3.3 大事务阻塞
**特征:**
- 主库有大操作
- 从库长时间执行同一事务

**处理:**
1. 确认大事务是否必要
2. 拆分大事务
3. 等待执行完成

### 3.4 锁等待
**特征:**
- 从库SQL线程等待锁
- show processlist显示Waiting

**处理:**
1. 识别阻塞的查询
2. 评估是否可以Kill
3. 联系应用团队

## 4. 处置决策

### 4.1 L3及以下延迟
- 持续监控
- 等待自动恢复
- 无需立即干预

### 4.2 L4延迟 (>5分钟)
1. 评估业务影响
2. 考虑是否需要主从切换
3. 准备应急预案
4. 上报技术负责人

### 4.3 复制中断
1. 立即停止写入主库
2. 检查错误原因
3. 修复复制
4. 验证数据一致性
5. 恢复服务

## 5. 后续处理

### 5.1 验证修复
```sql
-- 确认延迟已消除
SHOW SLAVE STATUS\G
-- 确认 Seconds_Behind_Master = 0
```

### 5.2 预防措施
- 优化大事务拆分
- 限制从库读压力
- 监控网络质量
- 定期检查复制健康

### 5.3 记录归档
- 记录故障时间线
- 记录处理过程
- 更新知识库案例
- 复盘改进
