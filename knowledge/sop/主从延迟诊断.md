# 主从延迟诊断 SOP

> 版本：v1.0 | 更新：2026-03-28 | 适用：MySQL/PostgreSQL主从复制延迟

---

## 1. 延迟判断标准

| 延迟时间 | 等级 | 说明 |
|----------|------|------|
| < 1s | 正常 | 无需处理 |
| 1-30s | 关注 | 观察原因 |
| 30s-5min | 警告 | 需要处理 |
| > 5min | 严重 | 立即处理 |
| ∞（中断） | 致命 | 立即处理 |

---

## 2. 排查步骤

### 2.1 第一步：确认延迟状态
```sql
-- MySQL
SHOW SLAVE STATUS\G
-- 关注：Seconds_Behind_Master, Slave_IO_Running, Slave_SQL_Running

-- PostgreSQL
SELECT * FROM pg_stat_replication;
-- 关注：pg_wal_lsn_diff(sent_lsn, write_lsn) AS write_diff
```

### 2.2 第二步：定位延迟原因

#### 原因A：从库IO瓶颈
```
症状：Slave_IO_Running=Yes, Seconds_Behind_Master持续增长
检查：
- 从库CPU/IO使用率
- 网络带宽占用
- 主库binlog传输速度
```

#### 原因B：大事务
```
症状：延迟突然增加，对应主库有大事务
检查：
- 主库最近执行的SQL
- binlog大小和数量
处理：拆分大事务，减少单事务大小
```

#### 原因C：从库负载高
```
症状：从库CPU/IO使用率接近100%
检查：
- 从库慢查询
- 从库复制SQL执行慢
- 从库IO性能
处理：优化从库性能，减少从库负载
```

#### 原因D：网络问题
```
症状：延迟断断续续，网络不稳定
检查：
- 主从网络延迟
- 网络带宽是否打满
- 网络抖动日志
```

#### 原因E：双1设置问题（MySQL）
```
症状：延迟持续，但从库负载不高
检查：
- innodb_flush_log_at_trx_commit
- sync_binlog
处理：评估是否可以调整为更安全的值
```

### 2.3 第三步：分析复制架构
```
检查项：
□ 从库数量
□ 复制链路（级联？）
□ 是否有复制过滤器（replicate-do-table）
□ GTID模式还是传统模式
```

---

## 3. 延迟影响评估

### 3.1 业务影响
```
评估维度：
1. 读写分离场景：从库延迟导致读取数据过期
2. 备份场景：延迟影响备份一致性
3. 容灾场景：延迟影响RPO
```

### 3.2 风险分级
| 场景 | 延迟容忍 |
|------|----------|
| 普通查询 | < 30s |
| 金融交易 | < 1s |
| 实时报表 | < 5min |
| 异步分析 | < 30min |

---

## 4. 处置方案

### 4.1 紧急处置（延迟>5min）
```
1. 确认是否有大事务在执行
2. 临时提高从库优先级
3. 如果是复制中断，参考复制中断SOP
4. 评估是否需要重启复制
```

### 4.2 短期优化
```
1. 优化从库IO性能
2. 调整MySQL双1参数（评估后）
3. 减少大事务
4. 考虑并行复制
```

### 4.3 长期优化
```
1. 升级从库硬件
2. 调整复制架构（改为半同步？改为并行复制？）
3. 数据归档减少数据量
4. 考虑延迟副本（延迟1小时，用于快速回滚）
```

## 5. 监控指标

| 指标 | 阈值 | 告警 |
|------|------|------|
| Seconds_Behind_Master | > 30s | 警告 |
| Slave_IO_Running | != Yes | 致命 |
| Slave_SQL_Running | != Yes | 致命 |
| 从库CPU使用率 | > 80% | 警告 |
| 从库IO使用率 | > 80% | 警告 |
