# 容量不足处理 SOP

> 版本：v1.0 | 更新：2026-03-28 | 适用：磁盘/表空间容量问题处理

---

## 1. 容量问题识别

### 1.1 容量指标
| 指标 | 关注阈值 | 告警阈值 | 紧急阈值 |
|------|----------|----------|----------|
| 表空间使用率 | > 75% | > 85% | > 95% |
| 磁盘使用率 | > 70% | > 80% | > 90% |
| 临时表空间 | > 70% | > 85% | > 95% |
| 归档表空间 | > 70% | > 85% | > 95% |

### 1.2 紧急场景
- 归档满：可能导致数据库HANG
- UNDO满：导致无法执行DML
- 临时表空间满：导致排序失败
- 系统表空间满：严重故障

## 2. 排查步骤

### 2.1 第一步：确认容量状态
```sql
-- 查看所有表空间使用率
SELECT tablespace_name, 
       round(used_percent, 2) AS used_percent,
       status
FROM dba_tablespace_usage_metrics;

-- 查看数据文件
SELECT file_name, tablespace_name, bytes/1024/1024 AS size_mb,
       autoextensible
FROM dba_data_files;
```

### 2.2 第二步：识别热点对象
```sql
-- 查看占用空间最大的表
SELECT owner, segment_name, segment_type, bytes/1024/1024 AS size_mb
FROM dba_segments
WHERE tablespace_name = '&TABLESPACE_NAME'
ORDER BY bytes DESC
FETCH FIRST 10 ROWS ONLY;

-- 查看最近增长最快的表
SELECT owner, table_name, num_rows, blocks, 
       last_analyzed
FROM dba_tables
WHERE owner NOT IN ('SYS', 'SYSTEM')
ORDER BY blocks DESC;
```

### 2.3 第三步：评估数据保留策略
```
问题确认：
1. 数据是否超过保留期限？
2. 归档数据是否可以清理？
3. 是否有测试数据可以删除？
4. 历史备份是否可以移走？
```

---

## 3. 处置方案

### 3.1 紧急扩容（立即执行）
```sql
-- 扩展表空间
ALTER TABLESPACE &TBS_NAME ADD DATAFILE 
  '/path/to/new.dbf' SIZE 1G AUTOEXTEND ON NEXT 100M MAXSIZE 10G;

-- 或增大现有数据文件
ALTER DATABASE DATAFILE '/path/to/datafile.dbf' RESIZE 10G;
```

### 3.2 数据清理（评估后执行）
```
清理优先级：
1. 临时文件/过期备份 → 无风险，立即清理
2. 历史归档数据 → 评估是否已备份，备份后可清理
3. 测试环境数据 → 确认无引用后删除
4. 业务历史数据 → 评估保留策略，迁移到归档表
```

### 3.3 归档处理
```sql
-- 归档旧数据（示例）
CREATE TABLE orders_archive AS 
SELECT * FROM orders 
WHERE create_time < SYSDATE - 365;

-- 删除原表历史数据（谨慎！）
DELETE FROM orders WHERE create_time < SYSDATE - 365;
COMMIT;

-- 重建表释放空间
ALTER TABLE orders MOVE;
```

## 4. 容量规划

### 4.1 增长率计算
```sql
-- 计算表空间增长率（每天）
SELECT tablespace_name,
       round((used_space - lag_used_space) / days_diff, 2) AS growth_per_day_mb,
       round((used_space - lag_used_space) / days_diff / 1024, 2) AS growth_per_day_gb
FROM (
  SELECT tablespace_name, used_space, 
         LAG(used_space) OVER (ORDER BY snapshot_time) AS lag_used_space,
         snapshot_time - LAG(snapshot_time) OVER (ORDER BY snapshot_time) AS days_diff
  FROM dba_tablespace_usage_history
  WHERE snapshot_time > SYSDATE - 30
);
```

### 4.2 扩容时间计算
```
剩余容量 / 日增长率 = 剩余天数

例如：
- 表空间剩余100GB
- 日增长率10GB
- 剩余天数 = 10天

结论：10天内需要扩容
```

## 5. 预防措施

| 措施 | 说明 |
|------|------|
| 容量监控 | 表空间>75%告警，>85%升级 |
| 自动扩容 | 开启autoextend，避免突发满 |
| 数据归档 | 定期归档历史数据 |
| 保留策略 | 明确定义数据保留周期 |
| 容量规划 | 每季度评估容量需求 |
