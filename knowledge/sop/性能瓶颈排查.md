# 性能瓶颈排查 SOP

> 版本：v1.0 | 更新：2026-03-28 | 适用：性能问题诊断

---

## 1. 性能问题识别

### 1.1 常见症状
| 症状 | 可能原因 |
|------|----------|
| 系统响应慢 | CPU高/IO高/锁等待 |
| 查询超时 | 慢SQL/锁/连接耗尽 |
| 会话堆积 | 慢SQL阻塞/死锁 |
| CPU使用率高 | 解析量大/排序/hash join |
| IO使用率高 | 全表扫描/热点数据 |

### 1.2 排查入口
```
优先确认问题类型：
□ 单一会话慢 → 查该会话SQL
□ 所有会话慢 → 查系统资源
□ 偶发慢 → 查Top SQL历史
□ 定时慢 → 查定时任务/统计信息收集
```

## 2. 系统资源排查

### 2.1 CPU瓶颈
```
检查步骤：
1. SELECT * FROM v$system_event WHERE event LIKE '%CPU%'  -- CPU等待
2. SELECT sql_id, elapsed_time, cpu_time FROM v$sql ORDER BY cpu_time DESC  -- Top CPU SQL
3. 查看OS层面的CPU使用率分布（user/sys/idle）

结论：
- CPU idle < 20% → CPU争用
- CPU idle > 80% 但应用慢 → 可能是应用端或网络问题
```

### 2.2 内存瓶颈
```
检查步骤：
1. SELECT * FROM v$sgastat WHERE pool = 'shared pool'  -- SGA使用
2. SELECT * FROM v$pgastat  -- PGA使用
3. 查看Buffer Cache命中率

结论：
- 命中率 < 95% → 可能需要调优
- 共享池碎片化 → 可能需要flush shared pool（谨慎）
```

### 2.3 IO瓶颈
```
检查步骤：
1. SELECT * FROM v$system_event WHERE event LIKE '%read%' OR event LIKE '%write%'
2. 查看Top SQL（按disk_reads/executions排序）
3. 查看数据文件IO统计

结论：
- IO等待高 → 存在大量物理读/写
- 单表全扫描 → 缺少索引
```

## 3. SQL级排查

### 3.1 慢SQL分析
```
第一步：获取慢SQL列表
SELECT sql_id, sql_text, elapsed_time, executions, disk_reads
FROM v$sql
WHERE last_active_time > SYSDATE - 1/24  -- 最近1小时
ORDER BY elapsed_time DESC;

第二步：查看执行计划
SELECT * FROM TABLE(dbms_xplan.display_cursor('&sql_id'));

第三步：分析执行计划
- 检查是否全表扫描（TABLE ACCESS FULL）
- 检查是否缺少索引
- 检查是否大量数据排序
- 检查join方式是否合理
```

### 3.2 锁阻塞分析
```
检查步骤：
1. 查看当前锁等待
SELECT * FROM v$lock WHERE block > 0;

2. 查看锁等待链
SELECT sid, serial#, username, event, blocking_session
FROM v$session
WHERE wait_class != 'Idle';

3. 分析阻塞者
- 阻塞者是何种SQL
- 是否长时间未提交
- 是否可以kill
```

### 3.3 会话分析
```
检查步骤：
1. 查看活跃会话
SELECT sid, serial#, username, status, sql_id, event, seconds_in_wait
FROM v$session
WHERE status = 'ACTIVE';

2. 查看等待事件
SELECT event, count(*) FROM v$session GROUP BY event;

3. 查看资源消耗TOP会话
SELECT sid, value AS cpu_time FROM v$sesstat WHERE statistic# = 12 ORDER BY value DESC;
```

## 6. 常见场景快速处理

### 场景1：偶发性能抖动
```
原因：统计信息过期 / 绑定变量嗅探
处理：
1. 重新收集统计信息
2. 固定执行计划（SPM）
```

### 场景2：定时性能差
```
原因：定时任务 / 统计信息收集 / 批量任务
处理：
1. 避开业务高峰期
2. 优化批量SQL
3. 分批执行
```

### 场景3：整体变慢
```
原因：数据量增长 / 缓存失效 / 系统资源不足
处理：
1. 确认是资源问题还是SQL问题
2. 分析Top SQL变化
3. 考虑扩容或归档
```
