# 连接数打满处理 SOP

> 版本：v1.0 | 更新：2026-03-28 | 适用：连接耗尽应急处理

---

## 1. 问题识别

### 1.1 症状
- 应用出现 `too many connections` 错误
- 新连接无法建立
- 数据库连接数接近 max_connections

### 1.2 紧急确认
```sql
-- 查询当前连接数
SELECT count(*) FROM v$session WHERE type = 'USER';

-- 查询最大连接数
SELECT value AS max_connections FROM v$parameter WHERE name = 'max_connections';

-- 查询各状态连接数
SELECT status, count(*) FROM v$session WHERE type = 'USER' GROUP BY status;
```

---

## 2. 快速分析

### 2.1 连接分布
```sql
-- 按用户统计连接
SELECT username, count(*) FROM v$session 
WHERE type = 'USER' GROUP BY username;

-- 按机器统计连接
SELECT machine, count(*) FROM v$session 
WHERE type = 'USER' GROUP BY machine;

-- 按程序统计连接
SELECT program, count(*) FROM v$session 
WHERE type = 'USER' GROUP BY program;
```

### 2.2 阻塞分析
```sql
-- 查看是否有锁导致连接阻塞
SELECT s.sid, s.serial#, s.username, s.status, 
       s.event, s.seconds_in_wait, b.sid AS blocker_sid
FROM v$session s
LEFT JOIN v$session b ON s.blocking_session = b.sid
WHERE s.type = 'USER' AND s.status = 'ACTIVE';
```

### 2.3 长时间空闲连接
```sql
-- 查询长时间空闲的连接（可安全kill的）
SELECT sid, serial#, username, machine, logon_time, 
       ROUND(SYSDATE - logon_time, 2) AS idle_days
FROM v$session
WHERE type = 'USER' 
  AND status = 'INACTIVE'
  AND last_call_et > 3600  -- 空闲超过1小时
ORDER BY last_call_et DESC;
```

## 3. 处置方案

### 3.1 L3处置（无需审批，可自动执行）
```
适用于：存在明显空闲连接可以释放
步骤：
1. 确认空闲连接的来源（用户名/机器/程序）
2. 优先联系应用负责人确认
3. Kill空闲会话（风险评估后）
4. 监控连接数变化
```

### 3.2 L4处置（需审批）
```
适用于：需要kill活跃会话
步骤：
1. 评估影响范围
2. 联系应用负责人
3. 获取审批
4. 选择性kill低优先级会话
5. 确认应用恢复
```

### 3.3 紧急扩容（需双重审批）
```
适用于：连接数持续打满，无法通过kill解决
步骤：
1. 评估max_connections调整影响
2. 申请双重审批
3. 临时提高max_connections
4. 安排后续扩容
5. 后续优化：连接池、应用连接管理
```

## 4. Kill会话操作规范

### 4.1 安全Kill步骤
```sql
-- 1. 先查看会话详情
SELECT sid, serial#, username, machine, program, status, logon_time
FROM v$session
WHERE sid = &sid;

-- 2. 确认可安全kill后执行
ALTER SYSTEM KILL SESSION '&sid,&serial#' IMMEDIATE;
```

### 4.2 Kill优先级
```
优先级1：INACTIVE + 长时间空闲（>30分钟）
优先级2：INACTIVE + 空闲（>10分钟）
优先级3：ACTIVE + 非核心业务
优先级4：ACTIVE + 核心业务（需审批）
优先级5：ACTIVE + 正在执行大事务（禁止kill）
```

### 4.3 禁止Kill的场景
- 正在执行大事务（DML涉及大量数据）
- 正在执行DDL
- 已知应用缺陷，kill会导致雪崩
- 核心业务高峰期

---

## 5. 根因分析与预防

### 5.1 常见根因
```
1. 连接泄漏：应用未正确关闭连接
2. 连接池配置过大：maxIdle配置不当
3. 慢SQL阻塞：单个慢SQL占用大量连接
4. 访问突增：活动/秒杀等场景
5. max_connections配置过小
```

### 5.2 预防措施
```
1. 应用层：确保连接正确关闭（finally块）
2. 应用层：合理配置连接池（maxIdle = 预期并发*0.8）
3. 数据库层：设置连接超时（idle_session_timeout）
4. 监控层：连接数超过80%告警
5. 架构层：读写分离，分散连接压力
```
