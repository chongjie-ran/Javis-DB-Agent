## 一、版本目标

| 维度 | 目标 |
|------|------|
| 稳定性 | 对抗性验证 + 自我合理化防护，减少"幻觉完成" |

## 三、6个Phase详细任务分解

### Phase 0: Hook生命周期系统（基座）

**目标**: 建立统一的生命周期钩子系统，是其他5个Phase的基座

**交付物**:
- `src/hooks/hook_system.py` — AgentHook基类 + CompositeHook组合
- `src/hooks/events.py` — Agent生命周期HookEvent枚举（与现有gateway/hooks/不同层级）
- `src/hooks/context.py` — HookContext数据类

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 0.1 | 设计HookEvent枚举（8个生命周期点） | 覆盖before_iter/after_iter/before_execute_tools/after_execute_tools/on_stream/on_error/on_complete/before_llm |
| 0.2 | 实现AgentHook抽象基类 | 8个方法默认pass，子类选择性override |
| 0.3 | 实现CompositeHook | 错误隔离（一个Hook失败不影响其他）+ 按优先级排序 |
| 0.4 | 实现HookContext | payload/iteration_count/token_count/error可访问 |
| 0.5 | 集成到现有gateway/hooks | 与现有HookEvent共存，AgentHook针对Agent层，现有Hook针对Tool层 |

**依赖**: 无

---

### Phase 1: AgentRunner重构

**目标**: 将现有Agent的执行循环重构为基于Hook的可扩展架构

**交付物**:
- `src/agents/agent_runner.py` — 新的AgentRunner类
- `src/agents/run_spec.py` — AgentRunSpec规范

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 1.1 | 设计AgentRunSpec | 包含goal/context/hooks/max_iterations/timeout/token_budget |
| 1.2 | 实现AgentRunner.run() | while循环+Hook调用骨架 |
| 1.3 | 迁移OrchestratorAgent | 使用新AgentRunner，验证行为一致 |
| 1.4 | 向后兼容 | 现有Agent基类不破坏，渐进式迁移 |
| 1.5 | 流式支持 | on_stream Hook正确传递LLM流式chunk |

**依赖**: Phase 0

---

### Phase 2: 双层记忆系统

**目标**: 解决长期运行中的Token膨胀和记忆混乱问题

**交付物**:
- `src/memory/short_term.py` — HISTORY.md管理（事件日志）
- `src/memory/long_term.py` — MEMORY.md管理（长期事实）
- `src/memory/consolidator.py` — Token超阈值时触发记忆整合
- `src/memory/llm_memory.py` — LLM主动记忆接口（save_memory工具）

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 2.1 | ShortTermMemory | 追加模式写HISTORY.md，支持grep查询 |
| 2.2 | LongTermMemory | 长期记忆的读写，LLM自主决定何时写入 |
| 2.3 | Token监控Hook | after_iteration检测token超阈值 |
| 2.4 | Consolidator | Token超阈值时调用LLM总结+压缩历史 |
| 2.5 | save_memory工具 | Agent可主动调用，保存关键信息到长期记忆 |

**Hook点**: `after_iteration` — 检测token_count是否超阈值

**依赖**: Phase 0

---

### Phase 3: ValidatorRegistry

**目标**: 提供"打破假设"的对抗性验证能力

**交付物**:
- `src/validation/validator_registry.py` — ValidatorRegistry核心
- `src/validation/validators/` — 内置validator集合
- `src/validation/breaking_validator.py` — BreakingValidator（主动构造边界条件）

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 3.1 | ValidatorRegistry核心 | register/validate/unregister + break()方法 |
| 3.2 | CompletionClaimValidator | 检测"完成声明"，触发对抗性验证 |
| 3.3 | BreakingValidator基类 | 主动构造边界条件，目标是"打破"而非"确认" |
| 3.4 | SQLBreakingValidator | 构造边界SQL（空值/超长/特殊字符）测试 |
| 3.5 | 集成到AgentRunner | after_iteration调用ValidatorRegistry |

**Hook点**: `after_iteration` — 检测完成声明，触发验证

**依赖**: Phase 0 + Phase 1

---

### Phase 4: 自我合理化防护

**目标**: 防止Agent自我合理化跳过质疑

**交付物**:
- `src/hooks/self_justification_guard.py` — 信号词检测Hook
- `src/hooks/challenge_injector.py` — 强制质疑注入机制

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 4.1 | 信号词库定义 | SELF_JUSTIFICATION_SIGNALS = 20+个信号词 |
| 4.2 | detect_self_justification() | 检测文本中是否包含信号词 |
| 4.3 | ChallengeInjector | 检测到信号词时，强制注入质疑prompt |
| 4.4 | 集成到after_iteration | 对LLM输出进行信号词扫描 |
| 4.5 | SOUL.md更新 | 道衍角色明确自我合理化防护意识 |

**Hook点**: `after_iteration` — 扫描LLM输出

**依赖**: Phase 0

---

### Phase 5: TaskScheduler

**目标**: 任务分类+并发隔离，最大化并行效率

**交付物**:
- `src/scheduler/task_classifier.py` — 任务分类器
- `src/scheduler/task_scheduler.py` — 任务调度器
- `src/scheduler/file_lock.py` — 文件锁（WRITE_SAME_FILE串行）

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 5.1 | 任务分类 | READ_ONLY/WRITE_SAME_FILE/WRITE_DIFF_FILE/VERIFY四类 |
| 5.2 | FileLock | 写同一文件时的串行锁 |
| 5.3 | TaskScheduler.execute() | 按类型调度，并行+锁分离 |
| 5.4 | 与AgentRunner集成 | before_execute_tools调用分类 |
| 5.5 | 调度策略配置化 | YAML/JSON可配置并行度 |

**Hook点**: `before_execute_tools` — 任务分类

**依赖**: Phase 0 + Phase 1

---

### Phase 6: Subagent双模式

**目标**: 探索模式（只读快速）和执行模式（精确可写）分离

**交付物**:
- `src/subagent/explore_spec.py` — ExploreSpec（timeout=300s, read-only）
- `src/subagent/execute_spec.py` — ExecuteSpec（timeout=3600s, writable）
- `src/subagent/subagent_runner.py` — Subagent执行器

**具体任务**:

| # | 任务 | 验收标准 |
|---|------|---------|
| 6.1 | ExploreSpec定义 | mode="explore", timeout=300, read_only=True |
| 6.2 | ExecuteSpec定义 | mode="execute", timeout=3600, read_only=False |
| 6.3 | SubagentRunner | 根据Spec类型选择模式，超时控制 |
| 6.4 | 与TaskScheduler集成 | Explore模式用READ_ONLY并行，Execute模式精细控制 |
| 6.5 | 成本追踪 | 记录explore vs execute的token消耗 |

**Hook点**: `before_iteration` — 模式切换

**依赖**: Phase 0 + Phase 5

## 五、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Hook系统与现有gateway/hooks冲突 | 高 | Phase 0.5专门做集成验证，分层清晰 |
| 记忆整合导致上下文丢失 | 中 | Consolidator有LLM参与，保证关键信息不丢失 |
| Validator误报阻断正常流程 | 中 | BreakingValidator设计为"建议"而非"阻止" |
| Subagent双模式增加复杂度 | 中 | Explore模式强制5分钟超时，防止漂移 |
